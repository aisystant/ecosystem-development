# Концепция автоматических отчётов ИИ 0.4

## 0. Назначение документа

Документ описывает систему автоматических отчётов, генерируемых ИИ по документам хранилища. Определяет: зачем нужны отчёты, для кого предназначены, как применять, и содержит полный каталог отчётов с их назначением.

---

## 1. Запуск генерации отчётов

### 1.1. Автоматическая генерация (GitHub Actions)

**Периодичность:** Еженедельно, **воскресенье в 08:00 по московскому времени** (05:00 UTC)

**Механизм:** GitHub Actions workflow `generate-reports.yml`

**Что происходит автоматически:**
1. Workflow запускается по расписанию
2. Генерируются все отчёты в папку `0.4`
3. Изменения автоматически коммитятся и пушатся в репозиторий
4. Результат доступен сразу в GitHub

### 1.2. Ручной запуск через GitHub Actions (рекомендуемый способ)

Для генерации отчётов с автоматическим сохранением в репозиторий:

1. Откройте репозиторий на GitHub
2. Перейдите во вкладку **Actions**
3. Выберите workflow **Generate AI Reports**
4. Нажмите **Run workflow**
5. Выберите опции:
   - `ai_analysis`: включить AI-анализ (требует ANTHROPIC_API_KEY)
   - `report_type`: выбрать конкретный отчёт или `all`
6. Нажмите **Run workflow**

Результат появится в репозитории автоматически после завершения workflow.

### 1.3. Ручной запуск локально с push в репозиторий

Если нужно сгенерировать отчёты локально и сразу отправить в репозиторий:

```bash
# 1. Убедитесь, что вы на актуальной версии main
git checkout main
git pull origin main

# 2. Сгенерируйте все отчёты
python3 .ops/build_report.py --report all

# 3. Проверьте, что файлы обновились
git status

# 4. Добавьте изменения
git add "content/0. Управление/0.4. Автоматические отчёты ИИ/"

# 5. Закоммитьте с сообщением
git commit -m "chore: обновлены автоматические отчёты ИИ

Сгенерированы отчёты:
- Архитектурный слепок хранилища
- Полнота хранилища
- Карта связей между документами
- Противоречия и несогласованности
- Терминологическая согласованность
- Рекомендации по развитию"

# 6. Запушьте в репозиторий
git push origin main
```

### 1.4. Генерация отдельных отчётов

```bash
# Архитектурный слепок
python3 .ops/build_report.py --report architecture-snapshot

# Полнота хранилища
python3 .ops/build_report.py --report completeness

# Карта связей
python3 .ops/build_report.py --report links-map

# Противоречия
python3 .ops/build_report.py --report contradictions

# Терминология
python3 .ops/build_report.py --report terminology

# Рекомендации
python3 .ops/build_report.py --report recommendations

# Все отчёты сразу
python3 .ops/build_report.py --report all
```

### 1.5. Генерация с AI-анализом

Для глубокого анализа с использованием Claude AI (требуется Anthropic API key):

```bash
# Установите переменную окружения
export ANTHROPIC_API_KEY="your-api-key"

# Запустите генерацию с AI-анализом
python3 .ops/build_report.py --report all --ai-analysis
```

Получить API ключ можно на https://console.anthropic.com/

## 2. Зачем нужны автоматические отчёты

### 2.1. Проблема

Хранилище экосистемы — живой организм: ~150 документов, 1.4 МБ текста, 9 семейств (F0–F9), 4 уровня иерархии. При таком объёме возникают проблемы:

| Проблема | Последствия |
|----------|-------------|
| **Потеря общей картины** | Трудно удержать в голове всю архитектуру |
| **Рассогласованность** | Документы противоречат друг другу |
| **Неполнота** | Пустые ячейки, отсутствующие связи |
| **Дрейф терминов** | Один термин используется в разных значениях |
| **Устаревание** | Документы не обновляются после изменений |

### 2.2. Решение

**Автоматические отчёты ИИ** — механизм непрерывной диагностики здоровья хранилища:
- Еженедельно формируют срез текущего состояния
- Выявляют проблемы до того, как они станут критичными
- Дают архитектору объективную картину без ручного анализа
- Превращают разрозненные документы в согласованную систему знаний

## 3. Для кого предназначены отчёты

| Роль | Основной отчёт | Как использует |
|------|----------------|----------------|
| **Архитектор экосистемы** | Архитектурный слепок, Содержательная полнота | Сверка видения с реальностью хранилища, проверка адресности документов |
| **Методист/Создатель** | Содержательная полнота, Карта связей, Терминология | Проверка своих документов на соответствие семейству и целевой аудитории |
| **Новичок команды** | Архитектурный слепок | Быстрое погружение в контекст |
| **ИИ-агенты** | Архитектурный слепок | Контекст для ответов |
| **DevOps/Техподдержка** | Технические противоречия | Исправление битых ссылок, дубликатов, метаданных |
| **Руководитель** | Рекомендации | Принятие решений о приоритетах |
| **Аудитор** | Все отчёты | Оценка зрелости документации |

### 4.3. Разделение проверок

| Тип проверки | Отчёт | Что проверяется |
|--------------|-------|-----------------|
| **Содержательные** | Содержательная полнота описания | Ошибки классификации по семействам, терминологические/логические/фактические противоречия |
| **Технические** | Технические противоречия | Дубликаты, битые wikilinks, некорректные метаданные, устаревшие версии |
| **Терминология** | Терминологическая согласованность | Дрейф терминов, неиспользуемые термины, кандидаты в глоссарий |

### 4.4. Краткое описание каждого отчёта

#### 1. Архитектурный слепок хранилища

**Для кого:** Архитектор экосистемы (еженедельное чтение)

**Что делает:** Формирует полное, структурированное описание всех содержательных идей хранилища. Не выявляет проблемы — только описывает «что есть». Архитектор сравнивает слепок со своим пониманием и сам видит расхождения.

**Ключевые разделы:**
- Миссия и ценности экосистемы
- Персоны и роли
- Целевая система (Созидатель)
- Архитектура платформы
- Подсистемы и их назначение
- Процессы и циклы
- Экономика и токеномика

#### 2. Содержательная полнота описания

**Для кого:** Архитектор, методисты

**Что делает:** Комплексный отчёт по содержательному качеству документов:
- Диагностирует заполненность матрицы документов по семействам F0–F9 и уровням
- Проверяет соответствие документов целевой аудитории семейства
- Выявляет содержательные противоречия (терминологические, логические, фактические, концептуальные)

**Ключевые разделы:**
- Матрица полноты по семействам
- Тепловая карта 3×3
- Список пустых документов
- Ошибки классификации по семействам
- Содержательные противоречия между документами
- Динамика заполнения

#### 3. Карта связей между документами

**Для кого:** Архитектор, методисты

**Что делает:** Строит граф зависимостей между документами на основе wikilinks. Выявляет изолированные документы, хабы, кластеры.

**Ключевые разделы:**
- Статистика связей
- Топ документов по связям
- Изолированные документы
- Кластеры по темам
- Mermaid-диаграмма графа

#### 4. Технические противоречия и несогласованности

**Для кого:** Архитектор, методисты, DevOps

**Что делает:** Выявляет технические проблемы структуры хранилища (НЕ содержательные — они в отчёте «Содержательная полнота»):
- Дубликаты документов
- Битые wikilinks
- Некорректные метаданные frontmatter
- Устаревшие версии и ссылки

**Ключевые разделы:**
- Полные дубликаты файлов
- Частичные дубликаты (>80% совпадения)
- Битые внутренние ссылки
- Ошибки метаданных
- Устаревшие версии документов

#### 5. Терминологическая согласованность

**Для кого:** Методисты, редакторы

**Что делает:** Анализирует использование терминов из глоссария. Выявляет: неиспользуемые термины, термины с дрейфом значений, слова-кандидаты в глоссарий.

**Ключевые разделы:**
- Покрытие терминов глоссария
- Термины с множественным использованием
- Кандидаты в глоссарий
- Термины с дрейфом значений

#### 6. Рекомендации по развитию

**Для кого:** Архитектор, руководитель

**Что делает:** Агрегирует данные из всех отчётов и формирует приоритизированный бэклог действий по улучшению хранилища.

**Ключевые разделы:**
- Оценка здоровья хранилища
- Критические действия (эта неделя)
- Важные улучшения (этот месяц)
- Бэклог улучшений
- Метрики прогресса

## 5. Как применять отчёты

### 5.1. Еженедельный цикл архитектора

```
Воскресенье 08:00 МСК → CI/CD генерирует отчёты
                              ↓
Понедельник утро → Архитектор читает «Архитектурный слепок»
                   → Сверяет со своим видением
                   → Отмечает расхождения
                              ↓
Понедельник/Вторник → Архитектор просматривает «Рекомендации»
                      → Создаёт задачи на неделю
                              ↓
В течение недели → Команда выполняет задачи
                   → PR → merge → обновление документов
                              ↓
Пятница → Ретро: что улучшили, какие метрики изменились
```

### 5.2. Использование по ситуациям

| Ситуация | Какой отчёт смотреть |
|----------|---------------------|
| «Хочу понять, что вообще есть в хранилище» | Архитектурный слепок |
| «Мой документ не связан с другими» | Карта связей |
| «Кажется, термин используется по-разному» | Терминология |
| «Мой документ адресован не той аудитории» | Содержательная полнота |
| «Что-то содержательно противоречит моему документу» | Содержательная полнота |
| «Битые ссылки или дубликаты» | Технические противоречия |
| «Какие документы нужно создать» | Содержательная полнота |
| «Что делать на этой неделе» | Рекомендации |
