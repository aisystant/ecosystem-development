# Архитектура знаний экосистемы

> **Назначение**: Этот документ фиксирует трёхуровневую модель организации знаний в экосистеме развития — разделение первых принципов, предметных эпистем и downstream-артефактов.

---

## 1. Проблема: смешение уровней знаний

В большинстве образовательных систем знания, методы их изложения и способы применения смешаны в неразличимую массу. Это приводит к:

- **Путанице знания с методом** — сама предметная область неотличима от способа её изложения
- **Невозможности переиспользования** — знание "захвачено" конкретным курсом или руководством
- **Устареванию формы вместе с содержанием** — обновление требует переделки всего
- **Искажению при адаптации** — каждая новая "сборка" изменяет исходную эпистему

---

## 2. Решение: трёхуровневая архитектура

Экосистема развития разделяет знания на **три уровня**:

```
┌─────────────────────────────────────────────────────────────┐
│  Уровень 1: ПЕРВЫЕ ПРИНЦИПЫ (FPF)                           │
│  ─────────────────────────────────────────────────────────  │
│  Мета-эпистема: правила корректного описания знаний.        │
│  Не является предметной областью.                           │
│  Задаёт язык различений для всех остальных уровней.         │
│                                                             │
│  Репозиторий: github.com/ailev/FPF                          │
│  Интеграция: 0.OPS/0.4.FPF-Integration/                     │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│  Уровень 2: ВТОРЫЕ ПРИНЦИПЫ (Episteme Packs)                │
│  ─────────────────────────────────────────────────────────  │
│  Эпистемы предметных областей: описание области             │
│  без дидактики, сценариев и маршрутов.                      │
│                                                             │
│  Примеры областей:                                          │
│  • Личное развитие (Personal Development)                   │
│  • Системная инженерия (Systems Engineering)                │
│  • Предпринимательство (Entrepreneurship)                   │
│                                                             │
│  Source-of-truth для своей области.                         │
│  ЖИВУТ В ОТДЕЛЬНЫХ РЕПОЗИТОРИЯХ.                            │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│  Уровень 3: DOWNSTREAM (Сборки и проекции)                  │
│  ─────────────────────────────────────────────────────────  │
│  Адаптации знаний под конкретные задачи:                    │
│  • Курсы и учебные программы                                │
│  • Руководства и чеклисты                                   │
│  • Персональные маршруты                                    │
│  • ИИ-представления и промпты                               │
│  • Эмбеддинги и индексы                                     │
│                                                             │
│  НЕ являются source-of-truth.                               │
│  Генерируются из Pack'ов, не наоборот.                      │
└─────────────────────────────────────────────────────────────┘
```

## 3. Уровень 1: Первые принципы (FPF)

### Что это

**First Principles Framework (FPF)** — мета-эпистема, задающая правила корректного описания любых предметных областей.

FPF определяет:
- Как различать практику, метод, инструмент, рабочий продукт
- Как не путать систему и процесс
- Как описывать область, не искажая её структуру
- Какие категориальные ошибки избегать

### Что FPF НЕ является

- **Не является предметной областью** — FPF не про "личное развитие" или "системную инженерию"
- **Не заменяется курсами** — FPF не "курс мышления", а правила построения курсов
- **Не является downstream** — FPF не руководство для пользователя

### Как используется в экосистеме

- FPF — фундамент для построения всех Episteme Packs
- Терминология S2R опирается на FPF
- Проверка корректности описаний — через FPF-паттерны

**Подробнее:** [[0.OPS/0.4.FPF-Integration/README.md]]

## 4. Уровень 2: Вторые принципы (SPF / Episteme Packs)

### Что это

**Second Principles Framework (SPF)** или **Episteme Pack** — описание конкретной предметной области, построенное на основе FPF.

Pack фиксирует:
- Характерные для области **различения** (distinctions)
- Ключевые **практики** (practices) с их методами и инструментами
- Типичные **рабочие продукты** (work products)
- Характерные **ошибки и заблуждения** области

### Что Pack НЕ содержит

- **Не содержит дидактики** — как объяснять, в каком порядке изучать
- **Не содержит сценариев** — какие упражнения делать, какие тесты проходить
- **Не содержит маршрутов** — какой путь проходить конкретному человеку
- **Не содержит адаптаций** — как адаптировать под аудиторию

### Реестр Pack'ов (SPF)

| Pack | Область | Репозиторий | Статус |
|------|---------|-------------|--------|
| **spf-personal** | Личное развитие, саморазвитие | [github.com/aisystant/spf-personal](https://github.com/aisystant/spf-personal) | Активный |
| **spf-systems-thinking** | Системное мышление | — | Планируется |
| **spf-entrepreneurship** | Предпринимательство | — | Планируется |

### Принцип: Pack — source-of-truth

Pack является **единственным источником истины** для своей области. Все downstream-артефакты (курсы, руководства, ИИ) должны:
- **Ссылаться** на Pack как на источник
- **Не изменять** содержание Pack при адаптации
- **Обновляться** при изменении Pack

---

## 5. Уровень 3: Downstream (Сборки и проекции)

### Что это

**Downstream-артефакты** — способы использования Pack'ов под конкретные задачи и аудитории.

### Типы downstream-артефактов

| Тип | Описание | Пример |
|-----|----------|--------|
| **Курсы** | Структурированное изложение с дидактикой | "Введение в системное мышление" |
| **Руководства** | Практические инструкции | "Как вести экзокортекс" |
| **Маршруты** | Персонализированные траектории | "Маршрут развития для менеджера" |
| **ИИ-представления** | Промпты, эмбеддинги | "System prompts для RouteGuide" |
| **Чеклисты** | Проверочные списки | "Чеклист системного подхода" |

### Принцип: Downstream — не source-of-truth

Downstream-артефакты:
- **Производные** — генерируются из Pack'ов
- **Множественные** — один Pack → много адаптаций
- **Изменяемые** — адаптируются под аудиторию
- **Не авторитетные** — при противоречии приоритет у Pack

## 6. Роль этого репозитория

### Что хранит этот репозиторий

1. **Архитектура экосистемы** — как устроена экосистема развития (ядра A, B, C, D)
2. **Правила организации знаний** — методология S2R, семейства документов
3. **Интеграция с FPF** — как экосистема использует первые принципы
4. **Указания на Pack'и** — где находятся Episteme Packs (внешние репозитории)
5. **Описания систем** — ИТ-платформа, агенты, цифровой двойник

### Что этот репозиторий НЕ хранит

1. **Сами Pack'и** — эпистемы предметных областей живут в отдельных репозиториях
2. **Курсы и программы** — downstream-артефакты обучения
3. **Руководства для пользователей** — downstream-артефакты применения
4. **Эмбеддинги и индексы** — технические downstream-артефакты

### Граница ответственности

```
ecosystem-development (этот репозиторий)
├── Роль: карта / конституция экосистемы
├── Описывает: ЧТО такое экосистема, КАК она устроена
├── Ссылается на: SPF/Pack'и как источники знаний
└── НЕ содержит: сами знания предметных областей

spf-personal (github.com/aisystant/spf-personal)
├── Роль: фреймворк вторых принципов для личного развития
├── Содержит: source-of-truth по содержанию области
├── Строится на: FPF как мета-уровне
└── НЕ содержит: обучение, маршруты, ИИ-артефакты

spf-* (другие репозитории Pack'ов)
├── Содержат: знания других предметных областей
├── Строятся на: FPF как мета-уровне
└── Используются: для генерации downstream

downstream-* (внешние репозитории/системы)
├── Содержат: курсы, руководства, ИИ-адаптации
├── Генерируются из: SPF/Pack'ов
└── Адаптируются под: конкретные аудитории и задачи
```

## 7. Как развивается архитектура

### Текущее состояние (2026-02)

- FPF интегрирован в `0.OPS/0.4.FPF-Integration/`
- Архитектура экосистемы описана в ядрах A, B, C, D
- **Первый SPF создан**: [spf-personal](https://github.com/aisystant/spf-personal) — фреймворк вторых принципов для личного развития

### Следующие шаги

1. **Развитие spf-personal** — наполнение содержанием области "Личное развитие"
2. **Определение формата SPF** — структура, метаданные, связи между Pack'ами
3. **Механизм синхронизации** — как downstream обновляется при изменении Pack
4. **Создание следующих SPF** — Systems Thinking, Entrepreneurship

### Принцип развития

Экосистема развивается "изнутри наружу":
1. Сначала — FPF (мета-уровень)
2. Затем — Pack'и (предметные области)
3. Затем — Downstream (адаптации)

Нельзя начинать с downstream и "извлекать" из него Pack.

## 8. Связь с терминологией FPF

Архитектура знаний использует терминологию FPF:

| Термин | FPF-определение | Применение в архитектуре |
|--------|-----------------|--------------------------|
| **Практика** | Повторяемая деятельность агента | Описывается в Pack как practice |
| **Метод** | Способ выполнения практики | Часть описания практики в Pack |
| **Инструмент** | Артефакт, используемый в методе | Упоминается в Pack, реализуется в downstream |
| **Рабочий продукт** | Результат практики | Описывается в Pack |
| **Область** | Domain — предметная область | Каждый Pack = одна область |

**Важно:** Не путать:
- **Область** (domain) ≠ **Система** (system) — область не "работает", система работает
- **Практика** ≠ **Процесс** — практика абстрактна, процесс конкретен
- **Знание** ≠ **Обучение** — знание описывает, обучение передаёт

## 10. Процесс развития SPF (стандарт для Pack-репозиториев)

Каждый SPF-репозиторий должен следовать единому процессу развития, чтобы не деградировать в "умную вики".

### 10.1. Фазы развития Pack'а

```
Фаза 0: Инфраструктура     → структура, guardrails, CLAUDE.md
     ↓
Фаза 1: Инициализация      → bounded context, базовые различения
     ↓
Фаза 2: Ingestion          → подключение внешних материалов
     ↓
Фаза 3: Анализ             → извлечение кандидатов в элементы Pack
     ↓
Фаза 4: Формализация       → интеграция в Pack через process lint
     ↓
Фаза 5: Итерация           → пересмотр при новом опыте
```

**Принцип:** Pack никогда не "завершается" — он эволюционирует.

### 10.2. Обязательная структура SPF-репозитория

Каждый Pack должен содержать:

```
spf-{область}/
├── CLAUDE.md              # Конституция работы ИИ с этим Pack
├── /pack/                 # Формализованное знание
│   ├── distinctions/      # Различения области
│   ├── practices/         # Практики с методами
│   ├── work-products/     # Рабочие продукты
│   └── failure-modes/     # Типичные ошибки
├── /process/              # Процесс создания знания
│   ├── ingestion.md       # Как обрабатывать материалы
│   └── lint.md            # Проверки корректности
├── /map/                  # Карта области (обязательный артефакт)
└── /inbox/                # Входящие материалы (временно)
```

### 10.3. Роль Claude в развитии SPF

**Claude — аналитик и формализатор, НЕ автор знаний.**

| Claude ДЕЛАЕТ | Claude НЕ ДЕЛАЕТ |
|---------------|------------------|
| Разбирает материал через процесс | Напрямую "пишет знания" |
| Извлекает различения, практики, ошибки | Копирует текст из источника |
| Предлагает кандидатов в Pack | Сохраняет нарратив |
| Прогоняет process lint | Добавляет сценарии/дидактику |
| Обновляет карту | Генерирует курсы |

### 10.4. Процесс ingestion (обработка материалов)

**Внешние материалы** (посты, заметки, руководства, кейсы):
- НЕ кладутся напрямую в Pack
- НЕ считаются знанием
- Используются ТОЛЬКО как вход для анализа

**Формат анализа материала** (что Claude должен выдать):

```
1. Distinctions extracted:
   - [список извлечённых различений]

2. Candidate practices:
   - [кандидаты в практики]

3. Candidate work products:
   - [кандидаты в рабочие продукты]

4. Candidate failure modes:
   - [кандидаты в типичные ошибки]

5. SoTA signals:
   - [что актуально / устарело / гипотеза]

6. Suggested changes to Pack:
   - [ничего / список файлов для изменения]
```

**Только после этого** — возможны изменения в Pack.

### 10.5. Guardrails (защита от деградации)

**Process lint** — обязательные проверки перед изменением Pack:

- [ ] Элемент получен через процесс анализа (не скопирован)?
- [ ] Терминология соответствует FPF?
- [ ] Нет дидактики, сценариев, маршрутов?
- [ ] Указан источник (материал, из которого извлечено)?
- [ ] Карта обновлена?

**Критерий успеха Pack'а:**
- Любой элемент можно объяснить через процесс его появления
- Downstream может использовать Pack без искажений
- Новые материалы **упрощают** структуру, а не усложняют
- Claude не может "написать что-то умное", не нарушив lint

## 11. Самопроверка

При работе с репозиторием проверяй:

- [ ] Описание не путает знание и способ его изложения?
- [ ] Предметная область не описывается как "система"?
- [ ] Downstream-артефакты не выдаются за source-of-truth?
- [ ] Ссылки на Pack'и указывают на внешние репозитории (когда будут созданы)?
- [ ] Терминология соответствует FPF?
