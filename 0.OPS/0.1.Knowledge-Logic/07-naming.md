# Правила именования (Naming Conventions)

## Принципы S2R именования

### 1. Используй реальные имена систем
**❌ Плохо**: `A.Target-System`, `B.Creation-System`

**✅ Хорошо**: `A.Созидатель`, `B.Экосистема-развития`

**Почему**: Абстрактные термины затрудняют понимание. Конкретные имена делают структуру самодокументируемой.

### 2. Используй дефисы для составных имён
**❌ Плохо**: `КомандыИСообщества`, `ИТПлатформа`

**✅ Хорошо**: `Команды-и-сообщества`, `ИТ-платформа-Экосистемы`

**Почему**: Дефисы улучшают читаемость, работают в URL и файловых системах.

### 3. Добавляй префиксы для позиций
**Формат**: `<Ядро><Позиция>.<Имя-системы>`

**Примеры**:
- `A1.Команды-и-сообщества` — надсистема в ядре A
- `B2.Экосистема-развития` — SoI в ядре B
- `C3.ИТ-служба` — система создания в ядре C

### 4. Роли именуй на английском
**Формат**: `<Система>/<Система>.<Позиция>.<Роль>`

**Роли**:
- `.1.Meaning` — смыслы
- `.2.Architecture` — архитектура
- `.3.Operations` — операции

**Примеры**:
- `A2.Созидатель/A2.1.Meaning/`
- `B2.Экосистема-развития/B2.2.Architecture/`
- `C3.ИТ-служба/C3.3.Operations/`

**Почему**: Английские названия ролей унифицированы в S2R методологии.

## Структура имён папок

### Корневые ядра
```
<Ядро>.<Имя-системы-SoI>/
```

**Примеры**:
- `A.Созидатель/`
- `B.Экосистема-развития/`
- `C.ИТ-платформа-Экосистемы/`

### Системы внутри ядра
```
<Ядро><Позиция>.<Имя-системы>/
```

**Позиции**:
- `1` — Надсистема (содержит SoI или задаёт контекст)
- `2` — Система интереса (SoI, центральная система)
- `3` — Система создания (создаёт SoI)

**Примеры**:
- `A1.Команды-и-сообщества/` (надсистема)
- `A2.Созидатель/` (SoI)
- `A3.Экосистема-развития/` (система создания)

### Роли внутри системы
```
<Ядро><Позиция>.<Роль>/
```

**Роли**:
- `.1.Meaning` — зачем система, ценности, миссия
- `.2.Architecture` — как устроена, модель, компоненты
- `.3.Operations` — как работает, процессы, практики

**Примеры**:
- `A2.1.Meaning/`
- `A2.2.Architecture/`
- `A2.3.Operations/`

### Метауровень
```
0.OPS/0.<Номер>.<Имя>/
```

**Примеры**:
- `0.OPS/0.1.Knowledge-Logic/`
- `0.OPS/0.2.Kernels-Bridge/`
- `0.OPS/0.5.AI-Reports/`

## Структура имён файлов

### Формат
```
<номер>-<название-через-дефисы>.md
```

**Компоненты**:
- `<номер>`: 01-99 (ведущий ноль для сортировки)
- `<название>`: kebab-case (слова через дефис)
- `.md`: расширение Markdown

### Нумерация документов

#### 01-09: Ключевые документы семейства
**Примеры**:
- `01-mission-statement.md` (F2: Meaning × SoI)
- `01-team-structure.md` (F6: Architecture × Система создания)
- `01-value-chain.md` (связи между ядрами)

#### 10-19: Дополнительные документы семейства
**Примеры**:
- `10-stakeholder-map.md`
- `11-use-cases.md`
- `12-personas.md`

#### 20-99: Специфичные документы
**Примеры**:
- `20-mcp-architecture.md`
- `25-ci-cd-pipeline.md`
- `30-onboarding-process.md`

### Специальные имена

#### README.md
**Назначение**: Обзор папки, навигация по содержимому

**Местоположение**: В каждой роли (*.1.Meaning/, *.2.Architecture/, *.3.Operations/)

**Пример содержимого**:
```markdown
# A2.1.Meaning

Документы о смыслах и ценностях созидателя:
- зачем созидать
- ценность созидания
- миссия созидателя

## Документы
- [[01-mission-statement.md]]
- [[02-value-proposition.md]]
```

#### О разделе *.*.md
**Старый формат** (устаревший): `О разделе 1.0.md`, `О разделе 2.0.md`

**Новый формат**: `README.md` в каждой роли

## Правила для подсистем

### Подсистемы в Architecture
**Формат**: `<Система>/<Система>.2.Architecture/<Имя-подсистемы>/`

**Примеры**:
- `B.Экосистема-развития/B2.Экосистема-развития/B2.2.Architecture/Сообщество/`
- `B.Экосистема-развития/B2.Экосистема-развития/B2.2.Architecture/ИТ-платформа/`

**Почему**: Подсистемы — часть архитектуры SoI, поэтому находятся в *.2.Architecture/

### Документы подсистем
Внутри папки подсистемы — обычные документы по тем же правилам:

```
B2.2.Architecture/Сообщество/
├── 01-community-model.md
├── 02-participation-tiers.md
└── README.md
```

## Примеры полных путей

### Документ в ядре A
```
A.Созидатель/
  A2.Созидатель/
    A2.1.Meaning/
      01-mission-statement.md
```

**Полный путь**: `A.Созидатель/A2.Созидатель/A2.1.Meaning/01-mission-statement.md`

### Документ в подсистеме ядра B
```
B.Экосистема-развития/
  B2.Экосистема-развития/
    B2.2.Architecture/
      ИТ-платформа/
        01-platform-architecture.md
```

**Полный путь**: `B.Экосистема-развития/B2.Экосистема-развития/B2.2.Architecture/ИТ-платформа/01-platform-architecture.md`

### Документ в метауровне
```
0.OPS/
  0.1.Knowledge-Logic/
    01-kernels-model.md
```

**Полный путь**: `0.OPS/0.1.Knowledge-Logic/01-kernels-model.md`

## Правила для ссылок

### Внутренние ссылки (Wikilinks)
**Формат**: `[[путь/к/документу.md]]` или `[[путь/к/документу.md|Текст ссылки]]`

**Примеры**:
```markdown
См. [[A2.Созидатель/A2.1.Meaning/01-mission-statement.md]]
См. [[01-kernels-model.md|модель ядер]]
```

### Относительные пути
**Предпочтительно**: Используй относительные пути от корня репозитория

**Примеры**:
```markdown
[[0.OPS/0.1.Knowledge-Logic/01-kernels-model.md]]
[[A.Созидатель/A2.Созидатель/A2.1.Meaning/01-mission-statement.md]]
```

## Frontmatter метаданные

### Обязательные поля
```yaml
---
family: F5  # Семейство документа (F1-F9)
kernel: B   # Ядро (A/B/C или F0)
system: B2  # Система (A1/A2/A3, B1/B2/B3, C1/C2/C3)
role: Architecture  # Роль (Meaning/Architecture/Operations)
status: draft  # Статус (draft/review/stable)
---
```

### Дополнительные поля
```yaml
---
# Обязательные
family: F5
kernel: B
system: B2
role: Architecture
status: stable

# Опциональные
depends_on:  # Документы, от которых зависит этот
  - A2.Созидатель/A2.2.Architecture/01-competency-model.md
  - C2.ИТ-платформа-Экосистемы/C2.2.Architecture/01-mcp-architecture.md
affects:  # Документы, на которые влияет этот
  - B2.Экосистема-развития/B2.3.Operations/01-platform-operations.md
tags:
  - platform
  - architecture
  - mcp
author: ИИ-агент SystemArchitect
updated: 2025-01-15
---
```

## Частые ошибки

### ❌ Абстрактные имена
```
A.Target-System/
B.Creation-System/
C.Subsystem/
```

### ✅ Конкретные имена
```
A.Созидатель/
B.Экосистема-развития/
C.ИТ-платформа-Экосистемы/
```

---

### ❌ Пробелы в именах файлов
```
01 Mission Statement.md
Platform Architecture.md
```

### ✅ Дефисы
```
01-mission-statement.md
01-platform-architecture.md
```

---

### ❌ Русские названия ролей
```
A2.Смыслы/
A2.Архитектура/
A2.Операции/
```

### ✅ Английские роли
```
A2.1.Meaning/
A2.2.Architecture/
A2.3.Operations/
```

---

### ❌ Неправильная структура подсистем
```
B.Экосистема-развития/
  Сообщество/  # Не в правильном месте!
  ИТ-платформа/
```

### ✅ Правильная структура
```
B.Экосистема-развития/
  B2.Экосистема-развития/
    B2.2.Architecture/
      Сообщество/  # Внутри Architecture!
      ИТ-платформа/
```

## Контрольный чеклист

При создании новой папки или файла проверь:

- [ ] Имя использует реальное название системы (не абстракцию)
- [ ] Составные имена через дефисы (`-`)
- [ ] Префикс ядра и позиции присутствует (`A1.`, `B2.`, `C3.`)
- [ ] Роль на английском (`.1.Meaning`, `.2.Architecture`, `.3.Operations`)
- [ ] Файлы нумеруются с ведущим нулём (`01-`, `02-`)
- [ ] Frontmatter содержит обязательные поля
- [ ] Путь соответствует семейству документа (F1-F9)
