# Семейства документов (Document Families)

## Концепция семейств

В S2R документы группируются в **9 семейств (F1-F9)** + мета-семейство **F0**, каждое из которых отвечает за определённую комбинацию "Система × Роль" в матрице 3×3.

**Важно:** Семейство — это «вид» на знания, а не физическая папка. Один документ относится к одному семейству, указанному в YAML Frontmatter.

**Критически важно для ИИ-анализа:** Каждое семейство имеет строго определённых **целевых читателей (заинтересованных лиц)**. Если документ в семействе адресован не той аудитории — это ошибка классификации, которую необходимо зафиксировать.

## Матрица семейств (3×3)

Семейства F1-F9 строятся на пересечении **уровней систем** (строки) и **проектных ролей** (столбцы). F0 — метасистема, стоящая над матрицей.

| Роль ↓ / Система → | Надсистема (1) | SoI (2) | Система создания (3) |
|---|---|---|---|
| **Meaning (1)** | F1 | F2 | F3 |
| **Architecture (2)** | F4 | F5 | F6 |
| **Operations (3)** | F7 | F8 | F9 |

### Альтернативное представление (по проектным ролям)

|                    | **Предприниматель** (Зачем?) | **Инженер** (Как устроено?) | **Менеджер** (Как работает?) |
|--------------------|------------------------------|-----------------------------|-----------------------------|
| **Надсистема**     | F1 — Видение и привлечение   | F2 — Контекст и интерфейсы  | F3 — Репутация и партнёрства |
| **Созидатель (SoI)**| F4 — Ценность для ученика   | F5 — Модель и компетенции   | F6 — Путь и сопровождение    |
| **Экосистема**     | F7 — Экономика и инвестиции  | F8 — Платформа и подсистемы | F9 — Команда и службы        |

## Принцип ролевой разработки документов

**Ключевая идея:** Во всех семействах F0-F9 документы должны разрабатываться определёнными **ролями-разработчиками** с применением **SoTA-методов** (State-of-the-Art) для удовлетворения информационных потребностей **заинтересованных лиц (читателей)**.

### Модель «Разработчик → SoTA-методы → Читатель»

```
┌─────────────────┐     ┌─────────────────┐     ┌─────────────────┐
│   РАЗРАБОТЧИК   │ ──► │   SoTA-МЕТОДЫ   │ ──► │    ЧИТАТЕЛЬ     │
│   (кто пишет)   │     │  (как пишет)    │     │ (для кого)      │
└─────────────────┘     └─────────────────┘     └─────────────────┘
```

**Почему это важно:**
1. **Качество контента** — применение SoTA-методов гарантирует профессиональный уровень документов
2. **Релевантность** — документ отвечает на реальные вопросы целевой аудитории
3. **Прослеживаемость** — понятно, кто отвечает за разработку и актуализацию
4. **Валидация** — можно проверить, использованы ли современные методы при создании

## Описание семейств (с примерами из наших ядер)

### F0: Метасистема и управление знаниями

**Главная роль:** держать всю картину знаний, онтологию и правила игры для людей и ИИ.

**Целевые читатели:**
- Администраторы хранилища знаний
- ИИ-агенты, работающие с документами
- Редакторы и библиотекари

**НЕ для:**
- Учеников (им не нужны правила классификации)
- Инвесторов (не интересует внутренняя онтология)
- Внешних партнёров

**Типичные документы:**
- Модель семейств и структура хранилища
- Стандарты оформления документов
- Глоссарий экосистемы
- Правила классификации
- Регламенты работы ИИ-агентов

**SoTA-методы:**
- Taxonomy Design
- Ontology Engineering
- Information Architecture

**ИИ-агенты:**
- Orchestrator — управляет другими агентами
- ConsistencyChecker — проверяет противоречия
- Librarian — помогает искать документы

### F1: Meaning × Надсистема (Видение и привлечение)

**Контекст**: Зачем нужна SoI для надсистемы? Какие ценности?

**Назначение:** описывает, зачем миру нужна экосистема, какой сдвиг она поддерживает.

**В наших ядрах**:
- **A1.1.Meaning/** — зачем команды нужны созидателю (культура, ценности)
- **B1.1.Meaning/** — тренды рынка, продвижение экосистемы
- **C1.1.Meaning/** — требования экосистемы к ИТ-платформе

**Целевые читатели:**
- Потенциальные участники экосистемы (первое касание)
- Широкая аудитория (публичные материалы)
- Маркетологи и продавцы
- Люди с проблемами, которые решает экосистема

**НЕ для:**
- Инвесторов (им нужна экономика → F7)
- Разработчиков (им нужна архитектура → F8)
- Наставников (им нужны методики → F5/F6)

**Типичные документы**: 
- Vision, Value Proposition для внешних стейкхолдеров
- Market Context
- Манифест экосистемы
- Проблемы и вызовы мира
- Целевые аудитории и сегменты
- JTBD-карты сегментов
- Стратегия маркетинга и продаж

**SoTA-методы:**
- Jobs-to-Be-Done (JTBD)
- Problem Discovery Interviews
- Market Ethnography
- Narrative Mapping
- Early Signal Testing

### F2: Meaning × SoI (Контекст и интерфейсы)

**Контекст**: Зачем существует SoI? Миссия, ценностное предложение.

**Назначение:** описывает, как созидатель встраивается в другие системы (сценарии использования).

**В наших ядрах**:
- **A2.1.Meaning/** — зачем созидать, ценность созидания
- **B2.1.Meaning/** — ценностное предложение экосистемы
- **C2.1.Meaning/** — зачем ИТ-платформа

**Целевые читатели:**
- Владельцы продукта (Product Owners)
- UX-дизайнеры и исследователи
- Пользователи, изучающие возможности системы

**НЕ для:**
- Инвесторов (им нужна экономика → F7)
- Внутренней команды разработки (им → F8)
- Партнёров по бизнесу (им → F3)

**Типичные документы**: 
- Mission Statement, Value Proposition, Vision
- Концепция использования созидателя
- Сценарии применения (User Stories, Job Maps)
- Контекстные диаграммы
- Внешние системы и интеграции
- Карты возможностей (Opportunity-Solution Tree)

**SoTA-методы:**
- Product Discovery (Teresa Torres)
- Opportunity-Solution Tree
- User Story Mapping
- Outcome-Driven Innovation (Ulwick)
- Service Blueprinting
- Behavioral Prototyping

**Назначение:** описывает, как экосистема работает с внешним миром.

**В наших ядрах**:
- **A3.1.Meaning/** — зачем создавать экосистему, бизнес-модели
- **B3.1.Meaning/** — экономика системы создания, инвестиции
- **C3.1.Meaning/** — миссия ИТ-службы

**Целевые читатели:**
- Партнёры и потенциальные партнёры
- Регуляторы и государственные органы
- Медиа и журналисты
- PR-менеджеры и коммуникаторы

**НЕ для:**
- Учеников (им нужна ценность и путь → F4/F6)
- Разработчиков (им нужна архитектура → F8)
- Инвесторов (им нужна экономика → F7)

**Типичные документы**: 
- Business Model, Investment Thesis, ROI
- Стратегия публичных коммуникаций
- Работа с партнёрами (модели партнёрства)
- Регуляторика и compliance
- Интересы заинтересованных лиц (внешних)
- Реестр обязательств перед внешними ролями

**SoTA-методы:**
- Reputation Management
- Public Narrative (Ganz)
- Community-Based Strategy
- Issue-Mapping & Conflict-Mapping
- Regulatory Foresight
- Crisis Communication Protocols

### F4: Architecture × Надсистема (Ценность и бизнес-модели)

**Контекст**: Структура надсистемы, контексты применения SoI.

**Назначение:** описывает ценность для созидателя и бизнес-модели.

**В наших ядрах**:
- **A1.2.Architecture/** — структура команд, роли, сообщества
- **B1.2.Architecture/** — контексты применения экосистемы (EdTech, Creator Economy)
- **C1.2.Architecture/** — как платформа интегрируется в экосистему

**Главный разработчик:** Визионер

**Целевые читатели:**
- Ученики и участники (что они получат)
- Потенциальные созидатели (принятие решения)
- Наставники (понимание ценности для учеников)

**НЕ для:**
- Инвесторов (им нужна экономика экосистемы → F7)
- Технических специалистов (им нужна архитектура → F8)
- Партнёров (им нужны условия → F3)

**Типичные документы**: 
- Stakeholder Map, Context Diagram, Use Cases
- Ценностное предложение для созидателя
- Этапы трансформации созидателя
- Бизнес-модели созидателя (как он будет зарабатывать)
- Продуктовые предложения (тарифы, программы)
- Описание того, что получает созидатель

**SoTA-методы:**
- Value Proposition Canvas
- Business Model Canvas
- Lean Canvas
- Customer Discovery

### F5: Architecture × SoI (Модель и компетенции)

**Контекст**: Архитектура SoI, компоненты, модель.

**Назначение:** описывает устройство созидателя как системы.

**В наших ядрах**:
- **A2.2.Architecture/** — модель созидателя, компетенции
- **B2.2.Architecture/** — платформа и подсистемы (Сообщество/, ИТ-платформа/)
- **C2.2.Architecture/** — архитектура ИТ-платформы (MCP, агенты, репозитории)

**Целевые читатели:**
- Методологи и проектировщики обучения
- Наставники (как устроена модель созидателя)
- Разработчики цифрового двойника

**НЕ для:**
- Учеников напрямую (им нужен путь → F6, ценность → F4)
- Инвесторов (им нужна экономика → F7)
- Маркетологов (им нужно привлечение → F1)

**Типичные документы**: 
- System Architecture, Component Diagram, Data Model
- Концепция созидателя (онтология)
- Модель созидателя (роли, состояния, способности)
- Карта компетенций
- Цифровой двойник (структура и функции)
- Уровни мастерства (Ученик → Интеллектуал → Профессионал → Исследователь → Просветитель)
- Траектории развития

**SoTA-методы:**
- Domain-Driven Design (DDD)
- Event Storming
- Architecture Decision Records (ADR)
- Systems Modeling Language (SysML)
- Model-Based Systems Engineering

## Межсемейные связи

### Вертикальные связи (внутри роли)
- **F1 → F2 → F3**: Каскад смыслов (Надсистема зачем → SoI зачем → Создание зачем)
- **F4 → F5 → F6**: Каскад архитектур (Контекст → Система → Команда)
- **F7 → F8 → F9**: Каскад операций (Внешние процессы → Использование → Создание)

### Горизонтальные связи (внутри системы)
- **F1 → F4 → F7**: Надсистема полностью (Зачем → Структура → Процессы)
- **F2 → F5 → F8**: SoI полностью (Зачем → Архитектура → Операции)
- **F3 → F6 → F9**: Система создания полностью (Зачем → Команда → Процессы)

### Диагональные связи (критические)
- **F2 (SoI зачем) ↔ F6 (Команда)**: Миссия SoI определяет структуру команды
- **F5 (Архитектура SoI) ↔ F9 (Процессы создания)**: Архитектура определяет процессы разработки
- **F3 (Экономика) ↔ F8 (Использование)**: Бизнес-модель определяет операционные процессы

## Правила проверки соответствия документа семейству

При анализе документа ИИ-агент ДОЛЖЕН проверить:

### Соответствие целевой аудитории

**Вопрос:** Для кого написан этот документ?

- Если документ в F1 адресован инвесторам → **ОШИБКА** (должен быть в F7)
- Если документ в F4 описывает архитектуру платформы → **ОШИБКА** (должен быть в F8)
- Если документ в F7 описывает путь ученика → **ОШИБКА** (должен быть в F6)

### Типичные ошибки классификации

| Документ содержит | Ошибочное семейство | Правильное семейство |
|-------------------|---------------------|----------------------|
| ROI, инвестиции, оценка компании | F1, F4 | F7 |
| Архитектура платформы, API, базы данных | F4, F6 | F8 |
| Путь ученика, онбординг, программы | F1, F7 | F6 |
| Партнёрские условия, регуляторика | F1, F4 | F3 |
| Модель компетенций, цифровой двойник | F4, F6 | F5 |
| Карта ролей команды, HR | F6, F7 | F9 |

### Формат фиксации ошибки

```markdown
**[ОШИБКА КЛАССИФИКАЦИИ]**
- Документ: {название}
- Текущее семейство: {F#}
- Обнаруженная целевая аудитория: {аудитория}
- Рекомендуемое семейство: {F#}
- Обоснование: {почему}
```

## Frontmatter для документов

Каждый документ должен содержать в YAML Frontmatter:

```yaml
---
family: F5                    # Семейство (F0-F9)
kernel: C                     # Ядро (A/B/C или F0 для 0.OPS)
system: C2                    # Система (A1/A2/A3, B1/B2/B3, C1/C2/C3)
role: Architecture            # Роль (Meaning/Architecture/Operations)
target_audience:              # Целевые читатели (ОБЯЗАТЕЛЬНО)
  - "Методологи"
  - "Наставники"
depends_on:                   # Родительские документы
  - "path/to/dependency.md"
affects:                      # Что зависит от этого документа
  - "path/to/affected.md"
status: draft | review | stable  # Статус документа
---
```
