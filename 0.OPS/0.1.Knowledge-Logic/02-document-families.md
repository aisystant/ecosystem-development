# Семейства документов (Document Families)

## Концепция семейств

В S2R документы группируются в **9 семейств (F1-F9)**, каждое из которых отвечает за определённую комбинацию "Система × Роль" в матрице 3×3.

## Матрица семейств

| Роль ↓ / Система → | Надсистема (1) | SoI (2) | Система создания (3) |
|---|---|---|---|
| **Meaning (1)** | F1 | F2 | F3 |
| **Architecture (2)** | F4 | F5 | F6 |
| **Operations (3)** | F7 | F8 | F9 |

## Описание семейств

### F1: Meaning × Надсистема
**Контекст**: Зачем нужна SoI для надсистемы? Какие ценности?

**В наших ядрах**:
- **A1.1.Meaning/** — зачем команды нужны созидателю (культура, ценности)
- **B1.1.Meaning/** — тренды рынка, продвижение экосистемы
- **C1.1.Meaning/** — требования экосистемы к ИТ-платформе

**Типичные документы**: Vision, Value Proposition для внешних стейкхолдеров, Market Context

### F2: Meaning × SoI
**Контекст**: Зачем существует SoI? Миссия, ценностное предложение.

**В наших ядрах**:
- **A2.1.Meaning/** — зачем созидать, ценность созидания
- **B2.1.Meaning/** — ценностное предложение экосистемы
- **C2.1.Meaning/** — зачем ИТ-платформа

**Типичные документы**: Mission Statement, Value Proposition, Vision

### F3: Meaning × Система создания
**Контекст**: Зачем создавать SoI? Экономика, инвестиции, мотивация.

**В наших ядрах**:
- **A3.1.Meaning/** — зачем создавать экосистему, бизнес-модели
- **B3.1.Meaning/** — экономика системы создания, инвестиции
- **C3.1.Meaning/** — миссия ИТ-службы

**Типичные документы**: Business Model, Investment Thesis, ROI

### F4: Architecture × Надсистема
**Контекст**: Структура надсистемы, контексты применения SoI.

**В наших ядрах**:
- **A1.2.Architecture/** — структура команд, роли, сообщества
- **B1.2.Architecture/** — контексты применения экосистемы (EdTech, Creator Economy)
- **C1.2.Architecture/** — как платформа интегрируется в экосистему

**Типичные документы**: Stakeholder Map, Context Diagram, Use Cases

### F5: Architecture × SoI
**Контекст**: Архитектура SoI, компоненты, модель.

**В наших ядрах**:
- **A2.2.Architecture/** — модель созидателя, компетенции
- **B2.2.Architecture/** — платформа и подсистемы (Сообщество/, ИТ-платформа/)
- **C2.2.Architecture/** — архитектура ИТ-платформы (MCP, агенты, репозитории)

**Типичные документы**: System Architecture, Component Diagram, Data Model

### F6: Architecture × Система создания
**Контекст**: Структура команды/инфраструктуры, создающей SoI.

**В наших ядрах**:
- **A3.2.Architecture/** — платформа, подсистемы (зеркало B2.2)
- **B3.2.Architecture/** — состав системы создания (ИТ-служба + другие команды)
- **C3.2.Architecture/** — состав ИТ-службы, DevOps-структура

**Типичные документы**: Team Structure, Org Chart, Toolchain Architecture

### F7: Operations × Надсистема
**Контекст**: Как надсистема взаимодействует с SoI? Репутация, партнёрства.

**В наших ядрах**:
- **A1.3.Operations/** — как работают команды, ритуалы
- **B1.3.Operations/** — репутация, партнёрства, PR
- **C1.3.Operations/** — SLA экосистемы к платформе

**Типичные документы**: Partnership Processes, Reputation Management, External Relations

### F8: Operations × SoI
**Контекст**: Как работает SoI? Процессы использования, путь пользователя.

**В наших ядрах**:
- **A2.3.Operations/** — путь созидателя, практики развития
- **B2.3.Operations/** — процессы экосистемы, сопровождение
- **C2.3.Operations/** — DevOps, мониторинг, эксплуатация платформы

**Типичные документы**: User Journey, Operating Procedures, Service Management

### F9: Operations × Система создания
**Контекст**: Как создаётся SoI? Процессы разработки, команда.

**В наших ядрах**:
- **A3.3.Operations/** — команда и службы (зеркало B3.3)
- **B3.3.Operations/** — процессы команды создания
- **C3.3.Operations/** — процессы ИТ-службы, SRE, CI/CD

**Типичные документы**: Development Process, CI/CD Pipelines, Team Rituals

## Межсемейные связи

### Вертикальные связи (внутри роли)
- **F1 → F2 → F3**: Каскад смыслов (Надсистема зачем → SoI зачем → Создание зачем)
- **F4 → F5 → F6**: Каскад архитектур (Контекст → Система → Команда)
- **F7 → F8 → F9**: Каскад операций (Внешние процессы → Использование → Создание)

### Горизонтальные связи (внутри системы)
- **F1 → F4 → F7**: Надсистема полностью (Зачем → Структура → Процессы)
- **F2 → F5 → F8**: SoI полностью (Зачем → Архитектура → Операции)
- **F3 → F6 → F9**: Система создания полностью (Зачем → Команда → Процессы)

### Диагональные связи (критические)
- **F2 (SoI зачем) ↔ F6 (Команда)**: Миссия SoI определяет структуру команды
- **F5 (Архитектура SoI) ↔ F9 (Процессы создания)**: Архитектура определяет процессы разработки
- **F3 (Экономика) ↔ F8 (Использование)**: Бизнес-модель определяет операционные процессы

## Правила именования документов

### Формат имени
```
<номер>-<название>.md
```

**Примеры**:
- `01-mission-statement.md` (F2)
- `02-system-architecture.md` (F5)
- `03-development-process.md` (F9)

### Нумерация
- **01-09**: Ключевые документы семейства (обязательные)
- **10-19**: Дополнительные документы семейства
- **20-99**: Специфичные/детальные документы

## Frontmatter для документов

Каждый документ должен содержать Frontmatter с метаданными:

```yaml
---
family: F5  # Семейство документа
kernel: C   # Ядро (A/B/C или F0 для 0.OPS)
system: C2  # Система (A1/A2/A3, B1/B2/B3, C1/C2/C3)
role: Architecture  # Роль (Meaning/Architecture/Operations)
status: draft | review | stable  # Статус документа
depends_on:
  - path/to/dependency.md
affects:
  - path/to/affected.md
tags:
  - tag1
  - tag2
---
```
