# Онтология S2R (S2R Ontology)

## Назначение документа

Этот документ определяет **фундаментальные сущности методологии S2R** и отношения между ними. Онтология — это формальное описание того, "что есть что" в S2R.

---

## Базовые сущности

### Система (System)

**Определение**: Целостная совокупность элементов, объединённых общей целью и взаимодействием.

**Атрибуты**:
- **Граница** — отделяет систему от окружения
- **Элементы** — составные части системы
- **Связи** — взаимодействия между элементами
- **Цель/Функция** — назначение системы

**Типы систем в S2R**:
1. **Надсистема (Supersystem)** — содержит SoI, задаёт контекст
2. **Система интереса (SoI)** — центральная система, которую описываем
3. **Система создания (Creation System)** — создаёт и поддерживает SoI

### Роль (Role)

**Определение**: Способ взаимодействия с системой, точка зрения на систему.

**Три роли в S2R**:

#### 1. Meaning (Смыслы)
- **Вопрос**: Зачем? Какова ценность?
- **Фокус**: Mission, Vision, Value Proposition
- **Позиция**: X.1.*

#### 2. Architecture (Архитектура)
- **Вопрос**: Как устроено?
- **Фокус**: Структура, компоненты, модель
- **Позиция**: X.2.*

#### 3. Operations (Операции)
- **Вопрос**: Как работает?
- **Фокус**: Процессы, практики, ритуалы
- **Позиция**: X.3.*

### Ядро (Kernel)

**Определение**: Полная 3×3 матрица "Системы × Роли", представляющая одну точку зрения на SoI.

**Структура ядра**:
```
Kernel = 3 системы × 3 роли = 9 папок

Системы:           Роли:
1. Надсистема      .1 Meaning
2. SoI             .2 Architecture
3. Создания        .3 Operations
```

**Атрибуты ядра**:
- **Имя** — реальное название SoI (не абстракция)
- **Префикс** — буква A/B/C/... для идентификации
- **SoI** — центральная система ядра (X2.*)

**Пример**:
```
A.Созидатель/
├── A1.Команды-и-сообщества/    # Надсистема
│   ├── A1.1.Meaning/
│   ├── A1.2.Architecture/
│   └── A1.3.Operations/
├── A2.Созидатель/              # SoI
│   ├── A2.1.Meaning/
│   ├── A2.2.Architecture/
│   └── A2.3.Operations/
└── A3.Экосистема-развития/     # Система создания
    ├── A3.1.Meaning/
    ├── A3.2.Architecture/
    └── A3.3.Operations/
```

### Документ (Document)

**Определение**: Единица знания в репозитории, описывающая один аспект системы.

**Обязательные атрибуты**:
- **family** — семейство F0-F9
- **kernel** — ядро A/B/C
- **system** — система X1/X2/X3
- **role** — роль Meaning/Architecture/Operations
- **target_audience** — целевые читатели

**Формат**: Markdown с YAML Frontmatter

## Отношения между системами

### ∋ (Содержит, Contains)

**Определение**: Надсистема физически или логически содержит SoI.

**Направление**: Надсистема ∋ SoI

**Свойства**:
- Транзитивность: если A ∋ B и B ∋ C, то A ∋ C
- Асимметричность: если A ∋ B, то B ∌ A

**Примеры**:
- Команды ∋ людей-созидателей
- Экосистема ∋ ИТ-платформу
- Сообщество ∋ участников

**Обратное отношение**: ∈ (входит в, belongs to)

### ⊃ (Включает как подсистему, Includes)

**Определение**: Составная система включает подсистемы как свои части.

**Направление**: Система ⊃ Подсистему

**Свойства**:
- Часть-целое (part-whole)
- Подсистемы могут взаимодействовать
- Целое не функционирует без критических подсистем

**Примеры**:
- Экосистема ⊃ (Сообщество, ИТ-платформа, Контент)
- Система создания ⊃ (ИТ-служба, Контент-команда, Бизнес-команда)

**Обратное отношение**: ⊂ (является подсистемой, is part of)

### → (Создаёт, Creates)

**Определение**: Система создания производит, разрабатывает, поддерживает SoI.

**Направление**: Система создания → SoI

**Свойства**:
- Каузальность: создатель предшествует созданному
- Зависимость: качество SoI зависит от системы создания

**Примеры**:
- ИТ-служба → ИТ-платформу
- Система создания Экосистемы → Экосистему
- Экосистема → Созидателя (через обучение/развитие)

**Обратное отношение**: ← (создаётся, is created by)

## Комбинированные отношения

### Цепочки систем

**Формат**: A ∋ B ← C ⊃ D

**Пример из нашей экосистемы**:
```
Команды-и-сообщества ∋ Созидатель ← Экосистема ⊃ (Сообщество, ИТ-платформа)
                                     ↑
                          Система создания Экосистемы ⊃ ИТ-службу
```

**Интерпретация**:
1. Команды содержат созидателей (физическое вхождение)
2. Экосистема создаёт созидателей (трансформация)
3. Экосистема включает сообщество и платформу (композиция)
4. Система создания включает ИТ-службу (часть команды)
5. Система создания создаёт экосистему (разработка)

## Правила композиции

### Правило 1: Один SoI на ядро

Каждое ядро описывает ровно одну систему интереса (позиция X2.*).

**✅ Правильно**:
- Ядро A: SoI = Созидатель (A2)
- Ядро B: SoI = Экосистема (B2)

**❌ Неправильно**:
- Ядро описывает две SoI одновременно

### Правило 2: Надсистема содержит SoI

Надсистема (X1) ВСЕГДА содержит (∋) SoI (X2).

**✅ Правильно**:
- A1 (Команды) ∋ A2 (Созидатель)
- B1 (Мир) ∋ B2 (Экосистема)

**❌ Неправильно**:
- Надсистема не содержит SoI
- SoI содержит надсистему

### Правило 3: Система создания создаёт SoI

Система создания (X3) ВСЕГДА создаёт (→) SoI (X2).

**✅ Правильно**:
- A3 (Экосистема) → A2 (Созидатель)
- B3 (Система создания) → B2 (Экосистема)

**❌ Неправильно**:
- SoI создаёт систему создания
- Надсистема создаёт SoI

### Правило 4: Реальные имена систем

Используй конкретные названия систем, а не абстракции.

**✅ Правильно**:
- `A.Созидатель/` (конкретная система)
- `B2.Экосистема-развития/` (конкретное название)

**❌ Неправильно**:
- `A.Target-System/` (абстракция)
- `B2.SoI/` (абстракция)

### Правило 5: Английские роли

Роли именуются на английском для универсальности.

**✅ Правильно**:
- `.1.Meaning/`
- `.2.Architecture/`
- `.3.Operations/`

**❌ Неправильно**:
- `.1.Смыслы/`
- `.2.Архитектура/`

## Мета-уровень F0

**Определение**: Уровень знаний, управляющий всеми ядрами.

**В репозитории**: Папка `0.OPS/`

**Отношение к ядрам**: F0 стоит "над" всеми ядрами, определяет правила игры.

**Не является ядром**, так как:
- Не имеет SoI (описывает методологию, а не конкретную систему)
- Не имеет 3×3 структуры (имеет произвольную структуру подпапок)
- Служит для управления и координации ядер

## Три ортогональных измерения знаний

S2R — это **формат структуры** (измерение "Форма"), а не уровень знаний. Знания в экосистеме организуются по **трём ортогональным измерениям**:

### Измерение 1: Содержание (что описывать)

Цепочка наследования терминологии: **FPF → SPF → Pack → Downstream**

| Уровень | Содержание |
|---------|------------|
| FPF | Мета-онтология (практика, метод, WP...) |
| SPF | Наследует FPF |
| Pack | Наследует SPF |
| Downstream | Наследует SPF (через Pack) |

### Измерение 2: Форма (как структурировать)

Форматы организации репозиториев:

| Формат | Когда применять |
|--------|-----------------|
| **S2R** | Несколько SoI, матричная организация, много связей |
| **Плоская структура** | Простой Pack с одной SoI |
| **Другие форматы** | По потребности |

**ВАЖНО:** S2R определяет ядра, матрицу 3×3, семейства F1-F9 — но НЕ определяет содержание или процесс производства знаний.

### Измерение 3: Процесс (как производить)

Определяется в SPF, применяется к Pack и Downstream:
- Ingestion — как обрабатывать материалы
- Process lint — проверки корректности
- Gates — критерии качества

### Таблица применения измерений по уровням

| Уровень | Содержание | Форма | Процесс |
|---------|------------|-------|---------|
| **1. FPF** | Мета-онтология | Свободная | — |
| **2. SPF** | FPF | Свободная | — |
| **3. Pack** | SPF | S2R или другой | Процесс SPF |
| **4. Downstream** | SPF (через Pack) | S2R или другой | Процесс SPF |

**Подробнее:** [[10-knowledge-architecture.md]]

## Семейства документов (F0-F9)

**Определение**: Классификация документов по матрице "Системы × Роли".

**Структура**:
```
         | Meaning (1) | Architecture (2) | Operations (3)
---------|-------------|------------------|---------------
Надсист. |     F1      |        F4        |       F7
SoI      |     F2      |        F5        |       F8
Создания |     F3      |        F6        |       F9
```

**Атрибуты семейства**:
- **Разработчик** — кто создаёт документы
- **Читатели** — целевая аудитория
- **SoTA-методы** — применяемые методологии

**Важно**: Семейство — это логическая группировка, не физическая папка.

## Валидация структуры

### Проверка ядра

Ядро валидно, если:
1. ✅ Имеет ровно 9 папок (3 системы × 3 роли)
2. ✅ X1 содержит (∋) X2
3. ✅ X3 создаёт (→) X2
4. ✅ Имя ядра = реальное название SoI
5. ✅ Роли именованы на английском

### Проверка документа

Документ валиден, если:
1. ✅ Имеет корректный Frontmatter с обязательными полями
2. ✅ family совпадает с позицией в матрице
3. ✅ target_audience совпадает с семейством
4. ✅ Находится в правильной папке (kernel/system/role)
5. ✅ Использует корректные символы отношений (∋, ⊃, →)
