# Принципы организации хранилища 0.1

## 0. Назначение документа

Этот документ фиксирует ключевые принципы организации хранилища знаний экосистемы. Принципы служат основой для принятия решений о структуре, процессах и инструментах работы с документами.

---

## 2. Основные принципы

### Принцип 1. Единственное место для каждого документа

Каждый документ имеет **одно правильное место** в хранилище.

**Это обеспечивает:**
- Отсутствие дубликатов
- Однозначность интерпретации
- Простоту поиска

**Как определить место:**
1. Определить систему (Мир / Созидатель / Экосистема)
2. Определить роль (Смыслы / Архитектура / Операции)
3. Поместить в соответствующую ячейку

---

### Принцип 2. Разделение типов информации

В хранилище **жёстко различаются типы документов:**

| Тип | Описание | Примеры |
|-----|----------|---------|
| Концептуальные | Визия, цели, ценности | Манифест, стратегия |
| Спецификации | Требования, архитектура | Описания систем, ADR |
| Процессные | Регламенты, инструкции | Онбординг, запуск потоков |
| Сырые | Черновики, заметки | Папка 0.9. Входящие |
| Автоматические | Отчёты ИИ | Папка 0.4 |

---

### Принцип 3. Человек редактирует ядро, ИИ — производные

**Человек редактирует:**
- Цели, визию, ограничения
- Ключевые понятия (глоссарий)
- Архитектурные решения верхнего уровня
- Критические организационные договорённости

**ИИ генерирует и обновляет:**
- Развернутые спецификации
- Чек-листы, тест-кейсы
- Агрегированные отчёты
- Карты связей

---

### Принцип 4. Каждый новый документ строится на существующем знании

Новый документ создаётся **только:**
- Из заранее определённого шаблона
- С автоподтяжкой терминов из глоссария
- Со ссылками на существующие документы

**Это обеспечивает:**
- Фрактальную согласованность
- Единый язык
- Отсутствие «изобретения велосипеда»

---

### Принцип 5. Автоматическая проверка каждого документа

Каждый документ при добавлении/обновлении проходит проверку:

1. **Статические проверки**
   - Формат, структура
   - Обязательные разделы
   - Ссылки не битые

2. **Семантические проверки ИИ**
   - Поиск противоречий
   - Несогласованность терминов
   - Нарушение уровней

3. **Проверка соответствия**
   - Правильная ячейка
   - Наличие YAML-шапки
   - Ссылки на родительские документы

---

### Принцип 6. Богатые Frontmatter

Каждый документ имеет Frontmatter:

**По содержанию:**
- Тип документа
- Уровень (ячейка)
- Зрелость (черновик / рабочий / утверждённый)

**По проектной сетке:**
- Семейство (F0-F9)
- Подсистема/компонент

**По целевым ролям (ОБЯЗАТЕЛЬНО):**
- `target_audience` — для кого предназначен документ
- `developer` — кто создаёт/редактирует документ

**По владению:**
- Кто создал
- Кто читает
- Кто имеет право редактировать

---

### Принцип 6.1. Каждый документ написан для конкретной аудитории

**Это критический принцип.**

Каждый документ в хранилище:
- Предназначен для **определённой целевой аудитории**
- Создаётся **определённой ролью (разработчиком)**
- **НЕ должен** быть адресован неправильной аудитории

**Семейство определяет аудиторию:**
- F0: ИИ-агенты, редакторы
- F1: Потенциальные участники
- F2: Пользователи продукта
- F3: Партнёры, медиа
- F4: Ученики, созидатели
- F5: Наставники
- F6: Ученики, наставники
- F7: Инвесторы, стратеги
- F8: DevOps, техлиды
- F9: Операторы, финансисты

**Несоответствие — критическая ошибка:**
- Документ в F1, адресованный инвесторам → должен быть в F7
- Документ в F4, описывающий архитектуру платформы → должен быть в F8

Подробнее см. [[Модель семейств документов 0.1]]

---

### Принцип 7. У каждого документа есть владелец

Для каждого документа определены:

- **Owner** — ответственный за актуальность
- **Maintainers** — могут принимать PR
- **Contributors** — могут предлагать изменения

Все изменения идут **только через PR**.

---

### Принцип 8. Единый источник правды (SSOT)

Для каждой сущности:
- Есть **один главный документ** — источник истины
- Остальные места — только ссылки и выдержки

При противоречии ИИ указывает на расхождение и предлагает синхронизацию.

---

### Принцип 9. Версионирование и трассируемость

Любое изменение документа:
- Имеет версию (commit)
- Имеет описание мотивации
- Позволяет ответить: кто, когда, зачем, какие последствия

---

### Принцип 10. Уровни формальности

Каждый документ имеет уровень зрелости:

| Уровень | Описание |
|---------|----------|
| F0 | Черновик, развёртывание мысли |
| F1 | Рабочий документ |
| F2 | Согласованный, есть чек-листы |
| F3+ | Формализованная спецификация |

---

### Принцип 11. Привязка к контексту

Каждый документ привязан к:
- Уровню системы (Надсистема / Целевая / Создания)
- Роли (Предприниматель / Инженер / Менеджер)
- Конкретному контексту (проект, версия)

---

### Принцип 12. Документ = артефакт + чек-лист

Каждый важный документ содержит:
- Структуру (шаблон)
- Содержание
- Микро-чек-лист принятия

---

### Принцип 13. Простой формат, сложный смысл

**Формат:** Markdown, YAML, JSON — простой и долговечный.

**Смысл:** Онтология, связи — в содержании и метаданных.

Это позволяет легко мигрировать между системами.

---

### Принцип 14. Регулярный рефакторинг

Хранилище — живая система:
- ИИ формирует отчёты о «долгах»
- Владельцы получают предложения по улучшению
- Дубликаты объединяются
- Гигантские файлы расщепляются

---

### Принцип 15. Безопасность и этика

Явно зафиксировано:
- Какие данные не попадают в репозиторий (секреты, ПД)
- Что ИИ не имеет права делать
- Различия между средами (dev / prod)

---

### Принцип 16. Связность документов

Каждый документ указывает:
- На документы уровня выше (контекст)
- На документы уровня ниже (детализация)
- На документы той же строки или столбца
