# Цепочка ценности (Value Chain)

## Общий обзор

Цепочка ценности показывает, как три ядра (A, B, C) связаны между собой и как создаётся ценность для всех участников экосистемы.

```
Мир (B1) → Команды (A1) → Созидатель (A2) → Экосистема (B2) → ИТ-платформа (C2)
  ↑            ↑               ↑                 ↑                    ↑
  │            │               │                 │                    │
  │            └───────────────┴─────────────────┘                    │
  │                  Система создания (B3) ⊃ ИТ-службу (C3) ─────────┘
  └─────────────────────────── Обратная связь ───────────────────────┘
```

## Уровень 1: От Мира к Созидателю

### Мир (B1) задаёт запрос
**Контекст**: Рынок труда, EdTech, Creator Economy

**Запрос**:
- Нужны квалифицированные разработчики
- Нужны системные мыслители
- Нужны предприниматели с инженерным бэкграундом

**Документы**:
- B1.1.Meaning: Тренды рынка, продвижение
- B1.2.Architecture: Контексты применения экосистемы
- B1.3.Operations: Репутация, партнёрства

### Команды и сообщества (A1) формируют среду
**Контекст**: Созидатель не работает в вакууме

**Ценность**:
- Команды предоставляют практический контекст
- Сообщества обеспечивают обмен опытом
- Менторы направляют развитие

**Документы**:
- A1.1.Meaning: Зачем команды, культура
- A1.2.Architecture: Структура команд, роли
- A1.3.Operations: Ритуалы, процессы команд

### Созидатель (A2) развивается
**Контекст**: SoI ядра A — человек

**Ценность**:
- Получает компетенции
- Создаёт проекты
- Развивает системное мышление

**Документы**:
- A2.1.Meaning: Зачем созидать, ценность
- A2.2.Architecture: Модель созидателя, компетенции
- A2.3.Operations: Путь развития, практики

**Связь с ядром B**: Созидатель развивается через экосистему (A3 = B2 из разных точек зрения)

## Уровень 2: От Созидателя к Экосистеме

### Экосистема развития (B2) предоставляет условия
**Контекст**: SoI ядра B — составная система

**Ценность для созидателя**:
- Платформа с инструментами
- Сообщество для обмена опытом
- Контент (курсы, руководства, знания)

**Подсистемы** (в B2.2.Architecture/):
- **Сообщество** ∋ созидателей (людей)
- **ИТ-платформа-Экосистемы** (техническая инфраструктура)

**Документы**:
- B2.1.Meaning: Ценностное предложение экосистемы
- B2.2.Architecture: Подсистемы (Сообщество/, ИТ-платформа/)
- B2.3.Operations: Процессы экосистемы, сопровождение

### Система создания Экосистемы (B3) разрабатывает
**Контекст**: Команда + процессы + инфраструктура, создающая B2

**Включает**:
- ИТ-службу (C3) — создаёт ИТ-платформу
- Контент-команду — создаёт курсы и материалы
- Команду сообщества — модерация, мероприятия
- Бизнес-команду — стратегия, экономика

**Документы**:
- B3.1.Meaning: Экономика, инвестиции
- B3.2.Architecture: Структура системы создания ⊃ ИТ-службу
- B3.3.Operations: Процессы разработки

**Связь с ядром C**: ИТ-служба (B3.2 ⊃ C3) создаёт ИТ-платформу (C2)

## Уровень 3: От Экосистемы к ИТ-платформе

### ИТ-платформа Экосистемы (C2) обеспечивает инфраструктуру
**Контекст**: SoI ядра C — техническая подсистема B2

**Ценность для экосистемы**:
- MCP-серверы (fsm-mcp, digital-twin-mcp, guides-mcp)
- ИИ-агенты (Orchestrator, RouteGuide, SystemArchitect и др.)
- Репозитории данных (Knowledge Repository, Digital Twin)

**Документы**:
- C2.1.Meaning: Зачем ИТ-платформа
- C2.2.Architecture: MCP, агенты, репозитории
- C2.3.Operations: DevOps, мониторинг, SRE

### ИТ-служба (C3) создаёт и поддерживает платформу
**Контекст**: Система создания в ядре C, подсистема B3

**Ценность**:
- Разработка новых фичей
- Поддержка существующих сервисов
- Мониторинг и SRE
- CI/CD и автоматизация

**Документы**:
- C3.1.Meaning: Миссия ИТ-службы
- C3.2.Architecture: Состав ИТ-службы, DevOps-структура
- C3.3.Operations: Процессы разработки, CI/CD

## Обратная связь (Feedback Loops)

### От созидателя к экосистеме
**Канал**: B2.3.Operations (процессы обратной связи)

**Данные**:
- Использование платформы (метрики)
- Запросы на фичи
- Отчёты об ошибках
- Предложения по улучшению

**Влияние**: B3 корректирует разработку на основе обратной связи

### От платформы к ИТ-службе
**Канал**: C2.3.Operations (мониторинг, SRE)

**Данные**:
- Метрики производительности
- Логи ошибок
- Инциденты
- Использование ресурсов

**Влияние**: C3 оптимизирует архитектуру и процессы

### От экосистемы к миру
**Канал**: B1.3.Operations (репутация, партнёрства)

**Данные**:
- Успехи выпускников
- Кейсы команд
- Партнёрства с компаниями
- Медиа и PR

**Влияние**: Мир признаёт ценность экосистемы, больше созидателей приходит

## Каскад зависимостей

### Смыслы (Meaning)
```
B1.1 (Тренды рынка)
  ↓
A1.1 (Зачем команды) ← B2.1 (Ценность экосистемы)
  ↓                        ↓
A2.1 (Зачем созидать) ← A3.1 (Бизнес-модель экосистемы)
  ↓                        ↓
                        B3.1 (Экономика системы создания)
                           ↓
                        C3.1 (Миссия ИТ-службы)
```

**Логика**: Миссия каждого уровня вытекает из ценности вышестоящего.

### Архитектура (Architecture)
```
B1.2 (Контексты применения)
  ↓
A1.2 (Структура команд) ← B2.2 (Подсистемы: Сообщество + ИТ-платформа)
  ↓                            ↓
A2.2 (Модель созидателя) ← C2.2 (Архитектура платформы)
  ↓                            ↓
A3.2 (Платформа и подсистемы) ← C3.2 (Состав ИТ-службы)
                              ↑
                          B3.2 (Структура системы создания ⊃ ИТ-службу)
```

**Логика**: Архитектура каждого уровня обеспечивает требования вышестоящего.

### Операции (Operations)
```
B1.3 (Репутация, партнёрства)
  ↓
A1.3 (Ритуалы команд) ← B2.3 (Процессы экосистемы)
  ↓                        ↓
A2.3 (Путь развития) ← C2.3 (DevOps, мониторинг)
  ↓                        ↓
A3.3 (Команда и службы) ← C3.3 (Процессы разработки)
                          ↑
                      B3.3 (Процессы системы создания)
```

**Логика**: Операции каждого уровня реализуют процессы вышестоящего.

## Ключевые точки интеграции

### Точка 1: Созидатель ↔ Команды
**Ядра**: A1 ↔ A2

**Отношение**: Команды ∋ созидателей

**Документы**:
- A1.2.Architecture/: Роли в командах
- A2.2.Architecture/: Компетенции для ролей

**Value Flow**: Созидатель получает роль в команде → применяет компетенции → команда создаёт ценность

### Точка 2: Созидатель ↔ Экосистема
**Ядра**: A2 ↔ B2 (A3)

**Отношение**: Экосистема создаёт созидателя (через развитие)

**Документы**:
- A2.3.Operations/: Путь развития
- B2.3.Operations/: Сопровождение пути

**Value Flow**: Созидатель проходит путь → получает компетенции → применяет в командах

### Точка 3: Экосистема ↔ ИТ-платформа
**Ядра**: B2 ↔ C2

**Отношение**: Экосистема ⊃ ИТ-платформу как подсистему

**Документы**:
- B2.2.Architecture/ИТ-платформа/: Требования от экосистемы
- C2.2.Architecture/: Архитектура платформы

**Value Flow**: Экосистема определяет требования → ИТ-платформа реализует → пользователи получают функциональность

### Точка 4: Система создания ↔ ИТ-служба
**Ядра**: B3 ↔ C3

**Отношение**: Система создания ⊃ ИТ-службу

**Документы**:
- B3.2.Architecture/: Структура системы создания
- C3.2.Architecture/: Состав ИТ-службы

**Value Flow**: Система создания координирует → ИТ-служба разрабатывает → платформа развивается

## Метрики ценности

### Уровень A (Созидатель)
**Метрики**:
- Количество компетенций
- Завершённые проекты
- Вклад в команды
- Траектория развития

**Источник**: A2.3.Operations/, digital-twin-mcp

### Уровень B (Экосистема)
**Метрики**:
- Количество активных созидателей
- Retention rate
- Engagement (активность в сообществе)
- Успешность выпускников

**Источник**: B2.3.Operations/, аналитика платформы

### Уровень C (ИТ-платформа)
**Метрики**:
- Uptime (доступность)
- Response time
- Количество инцидентов
- Deployment frequency

**Источник**: C2.3.Operations/, мониторинг SRE

## Приоритизация работы

### Правило 1: Ценность вверх
Любые изменения на уровне C должны увеличивать ценность для B, а через B — для A.

**Пример**: Новая фича MCP-сервера (C2) → улучшает работу агентов (B2) → ускоряет путь созидателя (A2)

### Правило 2: Обратная связь вниз
Проблемы уровня A должны транслироваться в изменения на уровнях B и C.

**Пример**: Созидатели не понимают рекомендации агента (A2.3) → улучшить логику RouteGuide (B2.2) → доработать fsm-mcp (C2.2)

### Правило 3: Согласованность ядер
Изменения в одном ядре должны согласовываться с другими.

**Пример**: Новая роль в команде (A1.2) → требует обновления модели созидателя (A2.2) → требует поддержки в Digital Twin (C2.2)
