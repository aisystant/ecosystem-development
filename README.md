> **Тип репозитория:** `Downstream/governance`

**Последнее обновление:** автоматически обновляется через GitHub Actions

## 🎯 О репозитории

Это хранилище знаний, построенное по методологии **S2R (Systems-Roles Repository)** — мульти-ядерной структуре для комплексного описания сложных систем.

Мы описываем:
- **A. Systems-Builder** — человека как целевую систему развития (создателя систем)
- **B. Aisystant-Ecosystem** — платформу + сообщество, создающую условия для развития
- **C. IT-Platform** — техническую инфраструктуру (MCP, агенты, репозитории)

```
                    World (B1)
                      │
        ┌─────────────┴─────────────┐
        │                           │
        │                  Teams-and-Communities (A1)
        │                           │
        │                      ∋ (содержит)
        │                           │
        ↓                           ↓
Aisystant-Ecosystem (B2) ←────── Systems-Builder (A2)
        │                      (создаётся)
        │
        ├─ ⊃ (включает подсистемы):
        │   ├─ Community ∋ созидателей
        │   └─ IT-Platform (C2)
        │
        ↑ (создаётся)
        │
Ecosystem-Builder (B3)
        │
        └─ ⊃ (включает):
            ├─ IT-Service (C3) → создаёт → IT-Platform (C2)
            └─ Другие команды
```

**Легенда**:
- `∋` — содержит физически (команды ∋ людей)
- `⊃` — включает как подсистему (экосистема ⊃ платформу)
- `→` — создаёт (ИТ-служба → платформу)

## 📚 Методология S2R

Репозиторий организован по методологии **Systems-Roles Repository (S2R)**.

**Подробнее о методологии:** [Документация S2R](https://github.com/TserenTserenov/s2r)

### Три уровня систем (в каждом ядре)
1. **Надсистема** (позиция 1) — содержит SoI, задаёт контекст
2. **Система интереса** (позиция 2) — центральная система, которую описываем
3. **Система создания** (позиция 3) — создаёт и поддерживает SoI

### Три роли (для каждой системы)
1. **Meaning** (.1.) — зачем система, ценности, миссия
2. **Architecture** (.2.) — как устроена, знания, компоненты
3. **Operations** (.3.) — как работает, процессы, практики

**Итого**: Каждое ядро = 3 системы × 3 роли = **9 папок**

## 📖 Архитектура знаний

> **Канонические определения FPF/SPF/Pack:** см. [SPF/docs/fpf-spf-pack.md](https://github.com/TserenTserenov/SPF/blob/main/docs/fpf-spf-pack.md)

### Четыре уровня знаний

```
┌─────────────────────────────────────────────────────────┐
│  Уровень 1: FPF (First Principles Framework)            │
│  Мета-онтология, язык различений                        │
│  Источник: github.com/ailev/FPF                         │
└─────────────────────────────────────────────────────────┘
                          │
                          ▼
┌─────────────────────────────────────────────────────────┐
│  Уровень 2: SPF (Second Principles Framework)           │
│  Фреймворк вторых принципов                             │
│  Задаёт форму и процесс для доменных знаний             │
└─────────────────────────────────────────────────────────┘
                          │
                          ▼
┌─────────────────────────────────────────────────────────┐
│  Уровень 3: PACK (Доменный source-of-truth)             │
│  Знание конкретной предметной области                   │
│  Пример: github.com/aisystant/spf-personal-pack              │
└─────────────────────────────────────────────────────────┘
                          │
                          ▼
┌─────────────────────────────────────────────────────────┐
│  Уровень 4: DOWNSTREAM (Производные представления)      │
│  Курсы, руководства, маршруты, ИИ-представления         │
│  Генерируются из Pack'ов, НЕ являются source-of-truth   │
└─────────────────────────────────────────────────────────┘
```

### Три ортогональных измерения

Помимо уровней, знания организуются по **трём измерениям**:

| Измерение | Что определяет | Примеры |
|-----------|----------------|---------|
| **Содержание** | Что описывать, терминология | FPF → SPF → Pack → Downstream |
| **Форма** | Как структурировать репозиторий | S2R, плоская структура |
| **Процесс** | Как производить знания | Ingestion, lint, gates (определяется в SPF) |

**S2R — это формат структуры (измерение "Форма"), а не уровень знаний.**

### Роль этого репозитория

```
ecosystem-development
├── Уровень: Downstream (4)
├── Формат: S2R (мульти-ядерная структура)
├── Роль: карта / конституция экосистемы
├── Описывает: ЧТО такое экосистема, КАК она устроена
└── НЕ содержит: сами знания предметных областей (Pack'и)
```

### Что это означает для нас

- Мы **НЕ source-of-truth** — изменения в знаниях должны идти через Pack
- Мы **основаны на Pack** — берём формализованное знание и создаём представления
- Форма (S2R) и процесс **свободные** — выбраны под нашу задачу

**ВАЖНО:** SPF ≠ Pack. SPF — фреймворк вторых принципов (форма + процесс). Pack — формализованное знание области (source-of-truth).

## 🛠️ Работа с репозиторием

### Для участников

**Изучение материалов:**
```bash
# Клонируй репозиторий
git clone https://github.com/aisystant/ecosystem-development.git
cd ecosystem-development

# Начни с ключевых документов
cat 0.OPS/0.1.Knowledge-Logic/01-kernels-model.md
```

**Навигация:**
- Используй wikilinks: `[[путь/к/документу.md]]`
- Ищи по тегам в Frontmatter
- Смотри зависимости (`depends_on`, `affects`)

### Для разработчиков

**Интеграция через MCP:**
```typescript
import { MCPClient } from '@modelcontextprotocol/sdk';

// Подключение к guides-mcp
const client = new MCPClient({
  server: 'guides-mcp',
  endpoint: 'http://localhost:3001'
});

// Получение документа
const guide = await client.call('getGuide', {
  kernel: 'A',
  system: 'A2',
  role: 'Architecture',
  document: '01-competency-model.md'
});
```

**Скрипты для работы:**
- `.ops/build_report.py` — генерация отчётов
- `.ops/classify_documents.py` — классификация документов по семействам
- `migrate_content.py` — миграция контента (уже выполнена)

### Для авторов контента

**Создание документа:**
1. Определи, к какому семейству относится документ (F1-F9)
2. Определи ядро (A/B/C) и систему (X1/X2/X3)
3. Определи роль (Meaning/Architecture/Operations)
4. Создай файл с правильным именем: `<номер>-<название>.md`
5. Добавь Frontmatter:

```yaml
---
family: F5  # Architecture × SoI
kernel: B
system: B2
role: Architecture
status: draft
tags:
  - architecture
  - platform
---
```

**Правила именования:**
- Используй реальные имена систем (не абстракции)
- Дефисы для составных имён: `Команды-и-сообщества`
- Роли на английском: `.1.Meaning`, `.2.Architecture`, `.3.Operations`
- Файлы с ведущим нулём: `01-`, `02-`
