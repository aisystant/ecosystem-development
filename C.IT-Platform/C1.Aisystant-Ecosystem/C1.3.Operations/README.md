# C1.3.Operations — SLA и операционные требования экосистемы

## Контекст

**Ядро:** C (ИТ-платформа)
**Система:** C1 — Экосистема развития (как надсистема для платформы)
**Роль:** Operations (Операции)
**Семейство:** F7

**Ключевой вопрос:** Как экосистема использует и управляет платформой?

## Назначение раздела

C1.3.Operations описывает **операционные требования** экосистемы к платформе:
- Какие SLA (Service Level Agreements) предъявляет экосистема?
- Как экосистема взаимодействует с платформой в ежедневных операциях?
- Какие процессы интеграции между экосистемой и платформой?

**Важно:** Это требования "сверху вниз" — от экосистемы к платформе. Внутренние процессы эксплуатации платформы описаны в C2.3.Operations.

## Типичное содержание документов F7 для C1

### SLA (Service Level Agreements)

| Метрика | Требование | Обоснование |
|---------|------------|-------------|
| **Availability** | 99.9% (8.76 часов простоя/год) | Ученики занимаются ежедневно |
| **Response Time** | < 500ms для 95% запросов | UX требования |
| **Data Recovery** | RPO < 1 час, RTO < 4 часа | Критичность данных учеников |
| **Support Response** | P1: 1 час, P2: 4 часа, P3: 24 часа | Ожидания пользователей |

### Операционные процессы

- Процедура запроса новой функциональности от экосистемы
- Процедура эскалации инцидентов
- Регулярные review между экосистемой и платформой
- Процесс согласования релизов
- Коммуникация о плановых работах

### Нагрузочные требования

- Ожидаемое количество пользователей (текущее и через год)
- Пиковые нагрузки (запуск потоков, дедлайны ДЗ)
- Сезонность (начало учебного года, каникулы)

## Пример SLA-документа

```yaml
# C1.3 Platform SLA
version: 1.0
effective_date: 2025-01-01

availability:
  target: 99.9%
  measurement_window: monthly
  exclusions:
    - planned_maintenance (с уведомлением за 72 часа)
    - force_majeure

performance:
  api_response_time:
    p50: 100ms
    p95: 500ms
    p99: 1000ms
  page_load_time:
    target: < 3s
    measurement: Core Web Vitals

support:
  p1_critical:
    response: 1 hour
    resolution: 4 hours
    examples: "Платформа недоступна, потеря данных"
  p2_high:
    response: 4 hours
    resolution: 24 hours
    examples: "Ключевая функция не работает"
  p3_medium:
    response: 24 hours
    resolution: 72 hours
    examples: "Неудобство, но есть workaround"

data:
  backup_frequency: hourly
  retention: 90 days
  rpo: 1 hour
  rto: 4 hours
```
