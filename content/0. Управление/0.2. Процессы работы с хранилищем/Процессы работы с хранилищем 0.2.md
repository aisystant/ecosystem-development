# Процессы работы с хранилищем 0.2

## 0. Назначение документа

Этот документ описывает полный цикл работы со всеми технологиями (Obsidian, GitHub, AI-ассистенты) и процесс внесения изменений в хранилище экосистемы через Pull Request.

---

## 1. Основные технологии

- **Obsidian** (локально, синхронизирован с Git): создание/редактирование всех текстов создателя экосистемы, быстрые связи, Dataview-реестры, шаблоны.    
- **GitHub**: версия Главного хранилища в виде Главного репозитория, а также другие репозитории, которые созданы на основе Главного; история решений, Pull Request'ы, автоматические проверки, релизы/экспорт.    
- **Cursor** или Claude: структурные правки и патчи (в т.ч. Markdown), «минимальные диффы» с коммитом в Git.    
- **AI-ассистенты**
- Технологии экономики сообщества         
- **Другие интерфейсы** (формы, таблицы, календари) — только как источники ссылок/вставок в Obsidian.

---

## 2. Основной цикл работы (7 шагов)

### Шаг 0. Подготовка  
Включены шаблоны, YAML-метаданные и теги. В `0.7 Классификация…` — правила «кто пишет/читает»:  
`происхождение` (человек/человек_и_ии/ии), `изменение` (только_человек/по_согласованию/автоматически), `читаемость` (человек/ии/оба), `доступ` (уровень).

### Шаг 1. Захват и разбор

- Человек пишет кратко во **"0.9 Входящее"** (мысли, ссылки, задачи).
    
- Каждая запись помечается YAML (тип, статус) и **перекладывается**:  
    — в **1. Идеи** (проблема/гипотеза/цель),  
    — в **3. Библиотека** (понятия/источники/программы),  
    — в **4. Системы** (если это изменяет архитектуру),  
    — или в **0.3 Решения** (если это принципиальный выбор).
    

### Шаг 2. Авторинг

- **Человек →** ключевые тексты управления: 0.0/0.2/0.3/0.4/0.6/0.7.    
- **Вместе с ИИ →** идеи/гипотезы (1._), понятия и программы (3._), карточки подсистем (4.*). ИИ предлагает правки/фрагменты, человек принимает/редактирует.    
- **Только ИИ →** черновые сводки, индексы (3.1 Картотека), агрегаты.
    

### Шаг 3. Связи и классификация

- В Obsidian проставляются вики-ссылки: идея ↔ система ↔ понятия.    
- YAML заполняется: `связи.системы/понятия/решения`, `статус`. Dataview-реестры обновляются сами.
    

### Шаг 4. Проверка качества (гейты)

- **Редактор** правит на ясность и единый стиль.    
- **Архитектор** сверяет согласованность систем (границы, входы/выходы).  
- **Проверяющий** собирает отчёт о состоянии: отражены ли принципы/решения в системах и планах; человек подтверждает или помечает несоответствия.    
- Если выбор принципиальный — фиксируется карточка в **0.3 Решения**.
    

### Шаг 5. Создание Pull Request и публикация в канонику

Описано ниже в разделе "Процесс внесения изменений".

### Шаг 6. Экспорт/распаковка

- Когда конкретный документ/набор зрелый, его путь назначается в «экспорт» (например, выведение отдельного репозитория систем/руководства).    
- GitHub выпускает релиз, другим ИТ-сервисам и агентам указывается, **откуда** брать канонику.
    

### Шаг 7. Мониторинг и обновления

- Раз в неделю ИИ пересобирает отчёты о состоянии.    
- Раз в спринт обновляется **0.4 Дорожная карта**: ближайший этап, рабочие продукты, бюджет времени.

### Кто что делает (матрица ролей)

| Узел                                             | Читает      | Пишет/меняет                                    | Инструменты                                |
| ------------------------------------------------ | ----------- | ----------------------------------------------- | ------------------------------------------ |
| **0.0/0.2/0.3/0.4/0.6/0.7/0.8** Управление       | человек, ИИ | **человек** (ИИ только предлагает формулировки) | Obsidian → PR GitHub                       |
| **0.1 Входящее**                                 | человек, ИИ | человек                                         | Obsidian                                   |
| __1._ Идеи/Гипотезы/Цели_*                       | оба         | **вместе** (ИИ-черновик, человек — редактор)    | Obsidian + ассистент «Стратег/Редактор»    |
| __2._ Продвижение_*                              | оба         | вместе                                          | Obsidian, ассистент «Редактор»             |
| __3._ Библиотека (Понятия/Программы/Картотека)_* | оба         | вместе (3.1 Картотека — **ИИ**)                 | Obsidian + Dataview + ассистент «Редактор» |
| __4._ Системы_*                                  | оба         | вместе (архитектура — человек ведёт)            | Obsidian / Cursor → PR GitHub              |

> Правило: всё, что **определяет курс/политику**, оформляет человек; всё, что **генерирует сводки/индексы/проверки**, делает ИИ; операционные тексты делаются **вместе** (ИИ предлагает, человек решает).

### Мини-ритм (день/неделя/спринт)

- **Ежедневно:** пополняю 0.9 «Входящее», разбрасываю по разделам; мелкие правки через Obsidian; PR по необходимости.    
- **Еженедельно:** ИИ собирает отчёты о состоянии; корректируем 0.3 «План работ».    
- **Каждый спринт:** обновляем 0.4 «Дорожная карта» (ближайший этап, рабочие продукты, бюджет времени); релизы/экспорт зрелых узлов.

## 3. Процесс внесения изменений через Pull Request

### 3.1 Участники процесса

| Роль | Описание | Права |
|------|----------|-------|
| **Контрибьютор** | Любой член команды | Предлагает изменения через PR |
| **Ревьюер** | Назначенный эксперт | Проверяет и комментирует |
| **Мейнтейнер** | Владелец раздела | Принимает или отклоняет PR |
| **ИИ-ассистент** | Claude/GPT | Помогает с оформлением и проверкой |

### 3.2 Пошаговый процесс внесения изменений

#### Вариант A: Новый документ

1. Создайте файл в папке `0.9. Входящие/`
2. Используйте шаблон из `0.3. Шаблоны/`
3. Заполните YAML-шапку
4. Опишите суть предложения

#### Вариант B: Изменение существующего документа

1. Создайте feature-ветку: `feature/<описание>`
2. Внесите изменения
3. Добавьте комментарий в YAML: `change_reason: "причина изменения"`

### 3.3 Создание Pull Request

```bash
# Создать ветку
git checkout -b feature/add-new-process

# Добавить изменения
git add .
git commit -m "feat: описание изменения"

# Отправить на GitHub
git push -u origin feature/add-new-process

# Создать PR через GitHub UI или CLI
gh pr create --title "Название изменения" --body "Описание..."
```

### 3.4 Автоматические проверки

GitHub Actions автоматически проверяет:
- [ ] Формат YAML-шапки
- [ ] Структура документа (обязательные разделы)
- [ ] Именование файла (`<Название> <N.M>.md`)
- [ ] Ссылки не битые
- [ ] Markdown-линтер

### 3.5 Ревью человеком

Ревьюер проверяет:
- [ ] Содержание соответствует ячейке матрицы 3×3
- [ ] Нет дублирования с другими документами
- [ ] Терминология из глоссария
- [ ] Ссылки на связанные документы

### 3.6 Обсуждение и доработка

1. Ревьюер оставляет комментарии
2. Контрибьютор вносит правки
3. Цикл повторяется до утверждения

### 3.7 Мёрж и интеграция

После утверждения:
1. Мейнтейнер делает merge в main
2. ИИ-ассистент обновляет индексы
3. Документ из `0.9. Входящие/` перемещается в целевую ячейку (если это новый документ)

## 4. Использование ИИ-ассистента

### 4.1 Подготовка PR с помощью ИИ

```
Промпт для Claude/GPT:

Я хочу добавить документ в хранилище экосистемы.

Тема: [описание темы]

Помоги мне:
1. Определить правильную ячейку (система + роль)
2. Выбрать название по правилам хранилища
3. Заполнить шаблон документа
4. Указать связанные документы
```

### 4.2 Автоматическое определение места

```
Промпт:

Проанализируй документ ниже и определи:
1. К какой системе он относится (Мир / Созидатель / Экосистема)?
2. Какой тип мышления (Смыслы / Архитектура / Операции)?
3. Итоговая ячейка?
4. Как его назвать?
5. Какие документы связаны?

---
[текст документа]
```

## 5. Особые случаи

### 5.1 Срочные изменения

Для критических правок (ошибки, безопасность):
1. Создать PR с меткой `urgent`
2. Уведомить мейнтейнера напрямую
3. Ревью в течение 4 часов

### 5.2 Масштабные изменения

Если изменение затрагивает много файлов:
1. Сначала создать issue для обсуждения
2. Получить согласие команды
3. Разбить на несколько PR

### 5.3 Изменения структуры

Изменения в `0.1. Логика хранилища и знаний/`:
1. Требуют согласования со всей командой
2. Фиксируются в ADR (`0.3. Архитектурные решения`)
3. Обновляется README.md

```markdown

- [ ] Исправление ошибки
- [ ] Изменение структуры

## Связанные документы

- [[Документ 1]]
- [[Документ 2]]

## Чек-лист

- [ ] YAML-шапка заполнена
- [ ] Название соответствует правилам
- [ ] Документ в правильной ячейке
- [ ] Добавлены ссылки на связанные документы
- [ ] Проверено на дублирование
```
