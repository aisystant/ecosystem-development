# Основные принципы организации хранилища: 
## 1. Принцип разных типов информации (слои артефактов)

В хранилище **жёстко различаются типы документов**:

1. **Концептуальные документы**  
    Визия, цели, ценность, контекст рынка/надсистемы, 3×3‑снимки, глоссарии, архитектурные решения верхнего уровня.  
    Это то, что формирует «как мы вообще думаем о системе».
    
2. **Системные спецификации**  
    Описания целевой системы и подсистем: требования, архитектура, интерфейсы, сценарии, модели данных и процессов.
    
3. **Документы системы создания**  
    Процессы разработки, обучения ИИ, эксплуатации, метрики флоу (WIP, циклы, релизы), роли людей.
    
4. **Первичная/сырая информация**  
    Черновые заметки, сырые результаты интервью/исследований, логи, телеметрия, эксперименты.
    
5. **Документы для ИИ/ИТ‑слоя**  
    Промпты, схемы агентов, политики, тест‑кейсы для проверки ИИ, обучающие выборки, спецификации API.
    
6. **Метадокументы**  
    Решения (ADR/DRR), обсуждения, протоколы, changelog‑и, чек‑листы, стандарты оформления.
    

Каждый файл относится **как минимум к одному типу**, и это отражено в метаданных (см. принцип 5).

---

## 2. Принцип «человек редактирует ядро, ИИ — производные»

Чёткое разделение ответственности:

- **Человек вручную редактирует**:
    
    - цели, визию, ограничения;
        
    - ключевые понятия (глоссарий, таксономии);
        
    - архитектурные решения верхнего уровня;
        
    - критические организационные договорённости (роли, ответственность, политики).
        
- **ИИ‑ассистенты генерируют и обновляют**:
    
    - развернутые спецификации на основе модельного ядра;
        
    - документацию к API, схемы БД, диаграммы;
        
    - чек‑листы, тест‑кейсы, рутинные описания процессов;
        
    - агрегированные отчёты, сводки, таблички.
        

Каждая сущность помечена флагом: **`editable_by = {human, ai, both}`**. Для `human‑only` действует правило: ИИ может **предлагать изменения**, но не коммитить без review через PR.

---

## 3. Принцип «каждый новый документ строится на уже существующем знании»

Никакого «пишем с нуля в вакууме»:

- Новый документ **создаётся только**:
    
    - из заранее определённого шаблона (для своего типа и 3×3‑ячейки), и
        
    - с автоподтяжкой уже имеющейся информации:
        
        - терминов из общего глоссария,
            
        - ссылок на существующие системы/подсистемы,
            
        - актуальных требований и решений.
            
- ИИ‑ассистент при создании документа:
    
    - показывает, **из каких артефактов** он собрал контекст (список источников),
        
    - явным списком фиксирует **зависимости** («этот документ согласован с X, Y, Z»).
        

Так обеспечивается «фрактальная согласованность»: локальная спецификация опирается на общий язык и общие решения, а не придумывает названия и смысл заново.

---

## 4. Принцип автоматической проверки каждого нового документа

Каждый документ при добавлении/обновлении проходит **автоматический пайплайн проверки**:

1. **Статические проверки**  
    Формат, структура, обязательные разделы, ссылки не битые, нет дубликатов имён/идентификаторов.
    
2. **Семантические проверки ИИ**
    
    - поиск противоречий с уже существующими документами;
        
    - несогласованность терминов (один термин — разные определения);
        
    - нарушение уровней: детальный документ не может противоречить более общему.
        
3. **Проверка соответствия шаблону и типу**  
    Правильная ячейка 3×3, правильный уровень детализации, наличие ссылок на родительские артефакты (над‑документы).
    
4. **Проверка прав доступа**  
    Разрешены ли автору изменения этого типа документа и конкретного файла.
    

PR не может быть слит, пока **проверки не зелёные**. Фейлы проверки — это не повод ругаться на людей, а **список подсказок**, что нужно поправить.

---

## 5. Принцип авто‑маркировки и богатых метаданных

Каждый документ автоматически (и частично вручную) получает **набор меток**:

- **По содержанию**:
    
    - тип документа (из п.1);
        
    - уровень: _общий / ячейка 3×3 / модуль / компонент_;
        
    - уровень формальности/зрелости (черновик, рабочий, утверждённый).
        
- **По проектной сетке**:
    
    - привязка к ячейке 3×3 (надсистема / целевая / создание × предприниматель / инженер / менеджер)
        
    - привязка к подсистеме/компоненту.
        
- **По ролям**:
    
    - кто создал (human / AI / pair);
        
    - кто читает (целевые аудитории/роли);
        
    - кто имеет право редактировать;
        
    - кто утверждает.
        
- **По ИИ‑аспекту**:
    
    - можно ли использовать как обучающие данные для ассистентов;
        
    - содержит ли чувствительные данные.
        

Часть метаданных ставится **автоматически** (по шаблону, по месту в дереве, по анализу текста), часть — **явно** при создании PR.

---

## 6. Принцип «у каждого документа есть владелец и правила изменения»

Для каждого документа определены:

- **`owner`** — ответственный человек/роль:
    
    - отвечает за актуальность содержания;
        
    - обязан реагировать на PR и issue, связанные с документом.
        
- **`maintainers`** — те, кто может:
    
    - принимать PR;
        
    - инициировать рефакторинг/расщепление документа.
        
- **`contributors`** — круг, кто может предлагать изменения (PR).
    

Все изменения идут **только через PR**, даже если правит владелец. Это даёт историю решений, review и автоматическую проверку.

---

## 7. Принцип «всё, что можно, — обучающий датасет для ИИ‑ассистентов»

Хранилище изначально проектируется как **учебная база для ИИ‑ассистентов платформы**:

- Все документы по умолчанию **оптимизируются для машинного чтения**:
    
    - структурированный текст (Markdown/Org + таблицы/JSON‑вставки);
        
    - чёткие заголовки, списки, чек‑листы;
        
    - явные определения понятий.
        
- Для обучения ИИ:
    
    - строятся выборки по типам документов (глоссарий, спецификации, решения и т.д.);
        
    - фиксируются связи «вопрос → ответ → использованные документы»;
        
    - PR и обсуждения (review‑комментарии) используются как примеры корректировки и уточнения.
        
- При этом есть слой **политик**:
    
    - что нельзя использовать для обучения (секретные, персональные, сырые данные без очистки);
        
    - как анонимизируется информация.
        

---

## 8. Принцип общей библиотеки понятий и общей информации

В хранилище есть **одна общая библиотека понятий**:

- Единый глоссарий (или несколько семейств, но с мостами между ними):  
    термины, определения, связи между понятиями.
    
- Единые **шаблоны** для типовых артефактов:
    
    - карточка узла 3×3;
        
    - спецификация системы/подсистемы;
        
    - описание процесса;
        
    - описание ИИ‑агента;
        
    - архитектурное решение/ADR.
        
- Любой документ вместо «своего» определения понятия **ссылается** на запись в глоссарии.  
    Если нужно новое понятие — сначала оно попадает в библиотеку, и только потом используется.
    

Это делает знание **однородным и сравнимым**: ИИ и люди видят одни и те же смыслы в разных местах.

---

## 9. Принцип единого источника правды (SSOT)

Для каждой сущности (понятие, система, подсистема, процесс, метрика):

- есть **один «главный» документ или блок**, который считается источником истины;
    
- остальные места в хранилище — только **ссылки, проекции, выдержки**, генерируемые ИИ.
    

Если где-то возникает противоречие, ИИ‑ассистент указывает:  
«Этот фрагмент расходится с источником X (строка Y). Предлагаю варианты синхронизации».

---

## 10. Принцип версионирования и трассируемости решений

- Любое изменение документа:
    
    - имеет версию (tag/commit);
        
    - имеет **описание мотивации** (микро‑ADR/DRR: зачем поменяли и какие последствия ожидаем).
        
- Всегда можно ответить на вопросы:
    
    - _кто_ принял решение;
        
    - _когда_;
        
    - _на основании каких данных_;
        
    - _какие документы зависят от изменённого_.
        

Идея — не просто хранить текст, а хранить **историю мышления и эволюции системы**.

---

## 11. Принцип уровней формальности и зрелости

Каждый документ имеет тег уровня, например:

- **F0** — черновые заметки / развёртывание мысли;
    
- **F1** — рабочий документ, понятный людям, но без формальных тестов;
    
- **F2** — согласованный документ, к которому привязаны чек‑листы и тест‑кейсы;
    
- **F3+** — формализованные спецификации, пригодные для автомата/машинной проверки.
    

Это позволяет:

- не требовать сразу формальности там, где идёт поиск;
    
- понимать, на что уже **можно опираться в коде и ИТ‑системах**, а что ещё «в разработке».
    

---

## 12. Принцип привязки к контексту (3×3 + bounded context)

Каждый документ:

- привязан к:
    
    - уровню системы (надсистема / ЦС / система создания);
        
    - роли (предприниматель / инженер / менеджер);
        
    - конкретному **контексту** (проект, продукт, версия стандарта).
        

Это:

- упрощает навигацию;
    
- снижает риск смешения разных смыслов в одном файле;
    
- позволяет ИИ делать локальные проверки: сначала согласовать документ внутри своего контекста, потом уже смотреть на мосты к другим.
    

---

## 13. Принцип «документ = мыслительный артефакт + чек‑лист»

Каждый важный документ:

- это не просто текст, а:
    
    - **структура** (шаблон);
        
    - **содержание**;
        
    - **микро‑чек‑лист принятия** (что должно быть выполнено, чтобы считать документ «ок»).
        

Это важно, чтобы ИИ мог:

- сам запускать эти чек‑листы при проверке;
    
- объяснять человеку, что именно не дотянуто (например, «нет метрик» или «нет привязки к 3×3»).
    

---

## 14. Принцип tool‑agnostic: формат прост, смысл сложен

Хранилище проектируется так, чтобы:

- **формат файлов** был простой и долговечный:
    
    - Markdown, текстовые таблицы, JSON/YAML для структур;
        
- **смысл** (онтология, связи) был в содержании и метаданных, а не в особенностях конкретного инструмента.
    

Это позволяет:

- легко подключать разные ИИ‑модели и утилиты;
    
- мигрировать между системами хранения;
    
- не зависеть от одного вендора/платформы.
    

---

## 15. Принцип регулярного рефакторинга хранилища

Хранилище рассматривается как живую систему, а не кладбище файлов:

- раз в N недель/месяцев ИИ‑ассистенты формируют **отчёт о «долгах»**:
    
    - дубликаты;
        
    - устаревшие документы;
        
    - гигантские файлы, которые надо расщепить;
        
    - места с наибольшим количеством противоречий.
        
- Владельцы документов получают предложения:
    
    - что объединить;
        
    - что разнести по нескольким файлам;
        
    - что перевести на более высокий уровень формальности.
        

---

## 16. Принцип безопасности и этики

Даже если проект открытый, надо явно зафиксировать:

- какие данные никогда не попадают в репозиторий (секреты, персональные данные);
    
- что ИИ‑ассистенты **не имеют права** делать (например, предлагать изменить юридически значимые договоры без участия юриста);
    
- как отличаются рабочие/песочницы/прод среда в части данных.

# Описание семейств документов 3×3 и ИИ‑слой

Экосистема описывается по матрице 3×3:

- **Вертикаль (уровни):**
    
    1. Надсистема (рынок/мир),
        
    2. Целевая система (экосистема как продукт),
        
    3. Система создания (организация/конвейер).
        
- **Горизонталь (роли):**  
    A) Предприниматель (ценность и стратегия),  
    B) Инженер (конструкция и качество),  
    C) Менеджер (процессы и эксплуатация).
    
    Знания для Системного аналитика
    

Из этих осей получаем 9 ячеек и добавляем **0‑е семейство** — метасистему управления знаниями. Итого 10 семейств: **F0–F9**.

Важно: **семейство — это не папка, а «вид» на знания**.  
Один и тот же файл может физически лежать, например, в `content/4. Системы/...`, но в YAML‑шапке иметь `family: F5` и участвовать в проверках/навигации как элемент архитектурного семейства.

## F0. Метасистема и управление знаниями

**Главная роль:** держать всю картину знаний, онтологию и правила игры для людей и ИИ.

### Назначение семейства

- Описывает **матрицу 3×3 и семейства F0–F9** как каноническую модель.
    
- Фиксирует **правила классификации документов**, уровни доступа, связь с concept‑graph.
    
- Определяет **регламенты работы ИИ‑агентов и людей**: кто что может менять, как защищаются ручные правки.
    
- Является точкой входа для **ConsistencyChecker**: что считать противоречием и когда блокировать merge.
    

### Типичные документы

**Общие:**

- Карта 3×3 и расшифровка ячеек (что означает каждая, какие метрики/ограничения).
    
- «Правила знания»: система классификации (`type`, `audience`, `edit_mode`, `layer`, `scope`, `security`, `family`, `cell3x3`).
    
- Онтология ключевых понятий (связь с `concept-graph`, FPF‑спека как источник).
    
- Регламент работы людей и ИИ‑агентов (роли, права, защита ручных правок).
    

**Детальные:**

- Шаблоны ВДВЧ, чек‑листы качества для разных типов артефактов.
    
- Формат метаданных документа (YAML‑шапка: `family`, `cell3x3`, `depends_on`, `affects`, `concepts` и т.п.).
    
- Правила эскалации при противоречиях: когда ИИ комментирует, когда блокирует merge, кто принимает решение.
    

### ИИ‑агенты семейства F0

- **F0.Orchestrator**
    
    - Смотрит на всё дерево документов и семейств.
        
    - Раздаёт задачи другим ИИ (architect, docs, critique, code и т.д.).
        
    - Описание: `content/agents/F0.Orchestrator.md`.

## Общие правила для всех семейств

### 1. Метаданные документа (YAML‑шапка)

В начале каждого Markdown‑файла в `content/`:

`--- id: DOC-F5-MEMORY-ARCH-001 title: Архитектура Memory Bank v1 type: doc audience: mixed edit_mode: mixed layer: architecture scope: global-core security: internal  family: F5 cell3x3: target-system.engineer  depends_on:   - DOC-F1-VISION-001   - DOC-F4-PORTFOLIO-001   - DOC-F0-ONTOLOGY-001 affects:   - SYS-MEMORY-BANK   - SYS-INTELLECT-TWINS  concepts: [Надсистема, Экосистема, Платформа, Интеллект] ---`

Минимум для включения в систему:

- `family` — основное семейство (F0–F9);
    
- `cell3x3` — явная привязка к ячейке;
    
- `depends_on` — родительские документы (обычно из F0/F1/F4/F5);
    
- `affects` — какие системы/метрики может сломать или изменить;
    
- `concepts` — связь с узлами `concept-graph`.
    

### 2. Граф знаний

- Концепты из `concept-graph` (Надсистема, Экосистема, Интеллект и т.д.) — **узлы графа**.
    
- Документы связываются с узлами через `concepts:`.
    
- На основе `family`, `depends_on`, `concepts` ИИ ищет:
    
    - какие документы описывают один и тот же смысловой узел;
        
    - нет ли противоречий между уровнями (F1/F4/F5 и т.д.).

## Проверка непротиворечивости (роль F0)

На каждый PR:

1. GitHub Actions запускает существующие скрипты:  
    `ops/classify_documents.py`, `ops/validate_classifications.py`, `ops/build_check_document.py`.
    
2. Дополнительный шаг `ops/check_consistency.py` (или `--ai-analysis`):
    
    - определяет изменённые документы (по `git diff`);
        
    - по полям `family`, `depends_on`, `concepts` подбирает контекст;
        
    - формирует prompt для **F0.ConsistencyChecker**.
        
3. ConsistencyChecker пишет отчёт в `artifacts/critique/<id>.md` с пометками `[OK] / [WARN] / [BLOCK]`.
    
4. Workflow:
    
    - зелёный чек, если только OK/WARN;
        
    - при BLOCK — требуется ручной review владельца соответствующего семейства.
        

Принцип: **ИИ не правит `content/` напрямую**, а работает только через `artifacts/` и человеческое review.
