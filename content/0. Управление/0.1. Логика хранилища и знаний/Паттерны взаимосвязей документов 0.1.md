# Паттерны взаимосвязей документов

## 0. Назначение документа

Этот документ описывает **типичные паттерны взаимосвязей** между документами в хранилище. Он помогает:
- Понять логику связей между документами
- Правильно использовать метаданные `depends_on`, `affects`, `tests`
- Выстраивать документы в соответствии с их ролями в знании

---

## 1. Основные типы связей

### 1.1. Зависимость (depends_on)

Документ **зависит** от другого, если использует его идеи, определения или решения.

**Пример метаданных:**
```yaml
depends_on:
  - "Проблемы и вызовы мира 1.1"
  - "Принципы организации хранилища 0.1"
```

**Когда использовать:**
- Документ опирается на концепции из другого документа
- Документ является детализацией идей из родительского документа
- Документ реализует стратегию, описанную в другом месте

### 1.2. Влияние (affects)

Документ **влияет** на другие документы, если изменения в нём требуют обновления зависимых документов.

**Пример метаданных:**
```yaml
affects:
  - "Принципы и философия экосистемы 3.1"
  - "Гипотезы развития экосистемы 3.1"
```

**Когда использовать:**
- Изменения в этом документе требуют ревизии других
- Документ задаёт направление для реализации
- Документ является источником требований

### 1.3. Проверка (tests)

Документ **проверяет** другой, если содержит гипотезы, метрики или критерии для валидации идей.

**Пример метаданных:**
```yaml
tests:
  - "Принципы и философия экосистемы 3.1"
  - "Видение и ответ экосистемы 1.1"
```

**Когда использовать:**
- Документ содержит проверяемые гипотезы о другом документе
- Документ описывает метрики для валидации идей
- Документ фиксирует критерии успеха реализации

### 1.4. Связь (related)

Документы **связаны**, если они относятся к одной теме, но не имеют прямой зависимости.

**Пример метаданных:**
```yaml
related:
  - "О разделе 1.0"
  - "Манифест экосистемы 1.1"
```

**Когда использовать:**
- Документы в одном семействе или разделе
- Документы на смежные темы
- Документы с перекрывающимся контекстом

## 2. Типичные паттерны взаимосвязей

### 2.1. Паттерн «Проблема → Решение → Реализация → Проверка»

**Описание:** Классическая цепочка от диагноза проблемы до проверки решения.

**Структура:**
```
Проблемы (диагноз)
    ↓ depends_on
Видение и ответ (ЗАЧЕМ)
    ↓ depends_on, реализуется через
Принципы и философия (КАК)
    ↓ depends_on, проверяется через
Гипотезы развития (ЧТО проверяем)
```

**Метаданные:**

1. **Проблемы и вызовы мира 1.1**
```yaml
affects:
  - "Видение и ответ экосистемы 1.1"
```

2. **Видение и ответ экосистемы 1.1**
```yaml
depends_on:
  - "Проблемы и вызовы мира 1.1"
affects:
  - "Принципы и философия экосистемы 3.1"
  - "Гипотезы развития экосистемы 3.1"
```

3. **Принципы и философия экосистемы 3.1**
```yaml
depends_on:
  - "Видение и ответ экосистемы 1.1"
affects:
  - "Гипотезы развития экосистемы 3.1"
  - "Экономическая модель экосистемы 3.1"
tests:
  - "Гипотезы развития экосистемы 3.1"
```

4. **Гипотезы развития экосистемы 3.1**
```yaml
depends_on:
  - "Видение и ответ экосистемы 1.1"
  - "Принципы и философия экосистемы 3.1"
tests:
  - "Принципы и философия экосистемы 3.1"
  - "Видение и ответ экосистемы 1.1"
```

**Пример из хранилища:**

| Документ | Семейство | Роль | Что делает |
|----------|-----------|------|------------|
| [[Проблемы и вызовы мира 1.1]] | F1 | Диагноз | Описывает проблемы мира на всех уровнях |
| [[Видение и ответ экосистемы 1.1]] | F1 | Видение | Отвечает на вопрос: ЗАЧЕМ миру экосистема |
| [[Принципы и философия экосистемы 3.1]] | F7 | Реализация | Описывает КАК устроена экосистема |
| [[Гипотезы развития экосистемы 3.1]] | F7 | Проверка | Определяет ЧТО проверяем и как измеряем |

**Визуализация:**
```
[Проблемы и вызовы мира 1.1]
         ↓ affects
[Видение и ответ экосистемы 1.1] ←---- depends_on
         ↓ affects
[Принципы и философия экосистемы 3.1] ← depends_on
         ↓ affects
[Гипотезы развития экосистемы 3.1] ←--- depends_on, tests
```

### 2.2. Паттерн «Стратегия → Тактика → Операции»

**Описание:** Каскад от стратегии к операционным деталям.

**Структура:**
```
Стратегический документ (ЗАЧЕМ и ЧТО)
    ↓ affects
Тактический документ (КАК)
    ↓ affects
Операционный документ (Детали ЧТО)
```

**Пример:**
```
Видение экосистемы 1.1
    ↓
Экономическая модель 3.1
    ↓
Токеномика и распределение 3.1
```

---

### 2.3. Паттерн «Концепция → Архитектура → Реализация»

**Описание:** От концептуального описания к техническим деталям.

**Структура:**
```
Концепция (ЗАЧЕМ и идея)
    ↓ affects
Архитектурное решение (КАК в общих чертах)
    ↓ affects
Техническая реализация (Детали КАК)
```

**Пример:**
```
Концепция ИИ-платформы 3.2
    ↓
Общая архитектура ИТ-платформы 3.2
    ↓
Операционная система ИИ-платформы 3.2
```

---

### 2.4. Паттерн «Общее → Специфичное»

**Описание:** От общего описания к конкретным случаям.

**Структура:**
```
Общий документ
    ↓ affects
Специфичный документ 1
Специфичный документ 2
Специфичный документ N
```

**Пример:**
```
Целевые аудитории 1.1
    ↓
JTBD-карты сегментов 1.1
    ↓
Маркетинговые материалы 1.1
```

---

### 2.5. Паттерн «Надсистема → Система → Подсистема»

**Описание:** Иерархия по уровням системности (соответствует структуре 3×3).

**Структура:**
```
Документ о надсистеме (раздел 1)
    ↓
Документ о системе (раздел 2)
    ↓
Документ о подсистеме создания (раздел 3)
```

**Пример:**
```
Проблемы и вызовы мира 1.1 (МИР)
    ↓
Концепция созидателя 2.2 (СОЗИДАТЕЛЬ)
    ↓
Принципы и философия экосистемы 3.1 (ЭКОСИСТЕМА)
```

## 3. Правила корректного использования связей

### 3.1. Принцип однонаправленности

**Правило:** Связи `depends_on` и `affects` всегда однонаправленные и противоположные.

**Хорошо:**
```yaml
# Документ A
affects:
  - "Документ B"

# Документ B
depends_on:
  - "Документ A"
```

**Плохо:**
```yaml
# Документ A
depends_on:
  - "Документ B"
affects:
  - "Документ B"  # Циклическая зависимость!
```

### 3.2. Принцип транзитивности

**Правило:** Если A → B → C, то явная связь A → C не обязательна (но допустима для важных случаев).

**Пример:**
```
Видение 1.1 → Принципы 3.1 → Экономическая модель 3.1
```

Связь `Видение 1.1 → Экономическая модель 3.1` не обязательна, но может быть добавлена в `related`.

### 3.3. Принцип семейственности

**Правило:** Документы из разных семейств (F1-F9) часто связаны через зависимости, документы внутри одного семейства — через `related`.

**Пример:**
- F1 (Мир × Предприниматель) → F7 (Экосистема × Предприниматель): `depends_on`, `affects`
- Документы внутри F7: `related`

### 3.4. Принцип проверяемости

**Правило:** Используйте `tests` только когда есть метрики, гипотезы или критерии проверки.

**Хорошо:**
```yaml
# Гипотезы развития 3.1
tests:
  - "Принципы и философия 3.1"  # Есть конкретные метрики для каждой гипотезы
```

**Плохо:**
```yaml
# Манифест 1.1
tests:
  - "Видение 1.1"  # Манифест не проверяет, а декларирует
```

## 4. Примеры правильного использования

### 4.1. Пример 1: Стратегические документы

**Проблемы и вызовы мира 1.1:**
```yaml
affects:
  - "Видение и ответ экосистемы 1.1"
```

**Видение и ответ экосистемы 1.1:**
```yaml
depends_on:
  - "Проблемы и вызовы мира 1.1"
affects:
  - "Принципы и философия экосистемы 3.1"
  - "Гипотезы развития экосистемы 3.1"
  - "Целевые аудитории 1.1"
related:
  - "Манифест экосистемы 1.1"
  - "О разделе 1.0"
```

### 4.2. Пример 2: Технические документы

**Концепция ИИ-платформы 3.2:**
```yaml
depends_on:
  - "Принципы и философия экосистемы 3.1"
affects:
  - "Общая архитектура ИТ-платформы 3.2"
  - "Карта подсистем 3.2"
related:
  - "О разделе 3.0"
```

**Общая архитектура ИТ-платформы 3.2:**
```yaml
depends_on:
  - "Концепция ИИ-платформы 3.2"
affects:
  - "Операционная система ИИ-платформы 3.2"
  - "Централизованное хранилище 3.2"
  - "Event-модель и интеграции 3.2"
```

### 4.3. Пример 3: Проверочные документы

**Гипотезы развития экосистемы 3.1:**
```yaml
depends_on:
  - "Видение и ответ экосистемы 1.1"
  - "Принципы и философия экосистемы 3.1"
tests:
  - "Принципы и философия экосистемы 3.1"
  - "Видение и ответ экосистемы 1.1"
related:
  - "Проблемы и вызовы мира 1.1"
  - "Экономическая модель экосистемы 3.1"
```

└──────────┬───────────────────────┘
           │ affects
           ↓
┌──────────────────────────────────┐
│ Видение и ответ экосистемы 1.1   │
│ (ЗАЧЕМ миру)                     │
└──────────┬───────────────────────┘
           │ affects
           ↓

│ Принципы и философия             │

           │ affects
           ↓

           ↑

## 6. Частые ошибки и их исправление

### 6.1. Циклические зависимости

**❌ Неправильно:**
```yaml
# Документ A
depends_on:
  - "Документ B"

# Документ B
depends_on:
  - "Документ A"
```

**✅ Правильно:**
```yaml
# Документ A (родительский)
affects:
  - "Документ B"

# Документ B (дочерний)
depends_on:
  - "Документ A"
```

### 6.2. Отсутствие обратных связей

**❌ Неправильно:**
```yaml
# Документ A
affects:
  - "Документ B"

# Документ B
# Нет depends_on!
```

**✅ Правильно:**
```yaml
# Документ A
affects:
  - "Документ B"

# Документ B
depends_on:
  - "Документ A"
```

### 6.3. Избыточные связи

**❌ Неправильно:**
```yaml
# Документ C
depends_on:
  - "Документ A"
  - "Документ B"  # B уже зависит от A, транзитивность
related:
  - "Документ A"  # Дублирование!
```

**✅ Правильно:**
```yaml
# Документ C
depends_on:
  - "Документ B"  # Только непосредственная зависимость
```
