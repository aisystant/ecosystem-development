

## 1. Назначение документа

Документ 0.1.2 задаёт стандарты, правила и требования к оформлению, структуре и содержанию всех документов репозитория. Он служит единой — как для людей, так и для ИИ — инструкцией, обеспечивающей:

- читаемость, понятность, консистентность;
    
- возможность автоматической классификации и валидации (ИИ-ассистентами);
    
- отсутствие дублирования и конфликтов;
    
- долгосрочную поддержку, версионирование и эволюцию знаний;
    
- единый стиль вне зависимости от автора.

## 2. Общие принципы оформления

### 2.1 Формат и технические требования

- Все документы — в формате Markdown (`.md`). Это облегчает отображение, diff-сравнение, версионирование и совместимость с Git/VCS. Это соответствует подходу «Docs as Code». ([apidog](https://apidog.com/blog/docs-as-code-best-practices/?utm_source=chatgpt.com "Docs as Code: Revolutionizing Software Documentation"))
    
- Имена файлов и папок — читаемые, предсказуемые, в **snake_case** или **kebab-case** (с дефисами), без специальных символов, без пробелов, без локальных символов там, где это может вызвать проблемы кодировок. Лучше придерживаться латиницы + дефис, если файл может быть использован внешними инструментами. Это правило упоминается в практике конвенций именования репозиториев/файлов. ([GitHub](https://github.com/GoldenbergLab/naming-and-documentation-conventions?utm_source=chatgpt.com "GoldenbergLab/naming-and-documentation-conventions"))
    
- Каждый документ должен иметь шапку с метаданными, в формате YAML-like или markdown-блоком, например:
    
    ```markdown
    ---
    title: "Название документа"
    author: "Имя / Ник"
    date_created: "YYYY-MM-DD"
    version: "v0.1"
    cell: "2.2"  # ячейка 3×3
    related: ["2.1", "3.2"]  # ссылки на смежные ячейки
    description: "Краткое описание содержания"
    ---  
    ```
    
    Это облегчает автоматическую обработку, индексацию, поиск, генерацию справок, таблицы содержания, а также работу ИИ-агентов.
    
- Документы должны быть разделены на смысловые блоки с использованием заголовков (`#`, `##`, `###` и т. д.), чтобы обеспечивать удобочитаемость и сканируемость. Ключевые моменты можно оформлять списками, таблицами, блоками цитирования, кодом, схемами. Согласно рекомендациям для документации, структура должна быть ясной, заголовки — логичными, и формат — единым. ([The GitHub Blog](https://github.blog/developer-skills/documentation-done-right-a-developers-guide/?utm_source=chatgpt.com "Documentation done right: A developer's guide"))
    
- Не перегружать документ «расширенными» темами: если раздел становится слишком большим, имеет смысл вынести его в отдельный документ и сослаться. Это позволяет избежать перегрузки, облегчает навигацию и поддерживает модульность. ([The GitHub Blog](https://github.blog/developer-skills/documentation-done-right-a-developers-guide/?utm_source=chatgpt.com "Documentation done right: A developer's guide"))
    

### 2.2 Стиль и язык

- Писать ясно и просто — избегать лишнего жаргона, если термин не определён в глоссарии. Любые специфичные термины давать в глоссарии (или ссылкой на него). Это повышает доступность документации для новых участников. ([The GitHub Blog](https://github.blog/developer-skills/documentation-done-right-a-developers-guide/?utm_source=chatgpt.com "Documentation done right: A developer's guide"))
    
- Каждый документ должен начинаться с краткого описания цели и применения: для кого, зачем, какие задачи решает.
    
- Не смешивать уровни абстракции: если документ на уровне стратегий/ценностей — не добавлять туда технические детали и операционные регламенты; для них — свои ячейки.
    

### 2.3 Версионирование и история изменений

- Каждый документ хранит метаданные `version` и `date_created` + `date_modified`. При изменениях — увеличивать версию (например, v0.1 → v0.2) и фиксировать даты.
    
- В теле документа — раздел `Changelog` (или `History`), где в виде списка записаны изменения: дата, автор, краткое описание изменений.
    
- При существенных изменениях, затрагивающих логику — делать заметку «Breaking change» и, при необходимости, создавать «legacy»-версию.
    

### 2.4 Связи между документами

- Каждый документ должен содержать ссылку на родительский контекст (вверху): например, если документ относится к ячейке 2.3 — обязательно упомянуть, что это «Операции Созидателя».
    
- В разделе «related» (метаданные) — перечислить связанные документы (соседи по строке, столбцу, уровню).
    
- В тексте — при упоминании идей/понятий/модулей, которые описаны в других документах — давать интрамные ссылки, чтобы навигация была удобной.
    
- Поддерживать целостность ссылочной структуры: если документ удалён/перенесён — ссылки обновлять; если меняется идентификатор ячейки — тоже.

## 3. Структура документа (шаблон)

Каждый документ, независимо от ячейки, должен иметь следующие разделы:

```markdown
# Название документа

## Цель и назначение

Кратко: для кого, зачем, какие задачи решает.

## Контекст и область применения

Где применяется, какие границы, предпосылки.

## Содержание / Основная часть

— Основные идеи  
— Модели, схемы, концепции  
— (При необходимости) Рабочие продукты: шаблоны, чеклисты, регламенты, схемы, модели  

## Связанные документы

Список документов, на которые опирается, и которые расширяет.  

## Changelog / История версий

| Дата       | Версия | Автор  | Изменения                         |
|------------|--------|--------|-----------------------------------|
| YYYY-MM-DD | v0.1   | …      | Создан документ                   |
| YYYY-MM-DD | v0.2   | …      | Обновлена архитектура, правки …   |

```

Такой шаблон обеспечивает единообразие, читаемость и машинную / человеческую обработку.

## 4. Процедуры и процессы для поддержки качества

### 4.1 Pull requests / ревью

- Все изменения документации — через Pull Request (или аналогичный механизм), с описанием изменений, обоснованием, при необходимости — ссылками на связанные документы. Это часть подхода «Docs as Code». ([apidog](https://apidog.com/blog/docs-as-code-best-practices/?utm_source=chatgpt.com "Docs as Code: Revolutionizing Software Documentation"))
    
- Рецензирование (peer-review) документов: перед мёржем документа — один или два ревьювера должны подтвердить, что документ соответствует стандартам 0.1.2.
    

### 4.2 Автоматическая проверка (CI)

По возможности — настроить CI, который при изменении документов:

- проверяет, что метаданные корректны (есть `title`, `cell`, `version`, `date_created` и т. д.);
    
- проверяет, что ссылки внутри документа не битые (интра-ссылки ведут на существующие файлы);
    
- проверяет формат Markdown, отсутствие синтаксических ошибок;
    
- (по желанию) проверяет, не дублирует ли документ уже существующий по названию + содержанию.
    

### 4.3 Регулярная ревизия и аудит

- Каждые N месяцев (например, квартал) — ревизия раздела 0.4 (отчёты ИИ). Проверка на устаревшие документы, неполноту, дубли, несогласованности.
    
- При изменении принципов (например, пересмотр структуры 3×3) — обновление этого стандарта + миграция затронутых документов.

## 5. Рекомендации по использованию

- При создании нового документа — сначала выбрать правильную ячейку 3×3, затем использовать шаблон из раздела 3.
    
- При внесении изменений — обновлять метаданные, версию, добавлять запись в `Changelog`.
    
- При ссылке на другой документ — использовать относительные пути, единый стиль ссылок, чтобы при переименовании / перемещении ссылок не ломались.
    
- При сомнениях — делать draft-версию, отмечать `draft: true` в метаданных; после финализации — удалять флаг.
