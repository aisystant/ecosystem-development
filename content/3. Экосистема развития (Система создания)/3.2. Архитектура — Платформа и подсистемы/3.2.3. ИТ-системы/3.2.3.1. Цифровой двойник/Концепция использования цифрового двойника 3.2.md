# Концепция использования цифрового двойника 3.2

## Цель и назначение

Документ описывает концепцию использования цифрового двойника (ЦД) для трёх ключевых процессов экосистемы развития:

1. **Изменение данных** — первичных и производных
2. **Проводник по персональному маршруту** — оценка ступени Ученика и выдача рекомендаций
3. **Генератор персонального руководства** — формирование индивидуальных руководств

**Адресаты:** архитекторы ИТ-платформы, команда разработки ИИ-ассистентов, методологи.

### 1.2. Первичные данные (1.*): источники и триггеры изменений

| Группа | Код | Источник | Триггер изменения |
|--------|-----|----------|-------------------|
| **Профиль** | 1.1.* | LMS, CRM | Регистрация, обновление профиля |
| **Курсы** | 1.2.*, 1.3.* | LMS | Старт/завершение курса, раздела, сессии |
| **Практики** | 1.4.* | LMS | Сдача задания, ответ на тест, создание модели |
| **Время** | 1.5.* | LMS, боты | Завершение помодоро, учебная сессия |
| **Финансы** | 1.6.* | Биллинг | Платёж, активация подписки, списание |
| **Квалификация** | 1.8.* | LMS, эксперты | Присвоение квалификации, сертификат |
| **Поиск** | 1.9.* | LMS | Каждый поисковый запрос |
| **Боты** | 1.10.* | Telegram, ИИ | Каждое сообщение боту/ИИ-ассистенту |
| **Клуб** | 1.11.* | Форум/клуб | Пост, комментарий, лайк, новая тема |

**Принцип записи первичных данных:**
- Запись происходит **синхронно** в момент события
- Каждое событие содержит: `timestamp`, `user_id`, `event_type`, `payload`
- Хранится **полная история** без удаления (append-only log)

### 1.3. Производные показатели (2.*): правила расчёта

| Показатель | Источники (1.*) | Формула расчёта | Частота обновления |
|------------|-----------------|-----------------|-------------------|
| **2.1.1 Часы в неделю** | 1.3.4 (сессии), 1.5.1 (помодоро) | SUM(длительность) за 7 дней | Ежедневно |
| **2.1.2 Доля дней со слотом** | 1.3.4, 1.5.1 | COUNT(дней с активностью) / 7 | Ежедневно |
| **2.1.6 Индекс регулярности** | 2.1.1, 2.1.2, история | Weighted(часы, регулярность, стабильность) | Еженедельно |
| **2.2.1 Интенсивность практики** | 1.4.2 (ответы), 1.4.3 (кейсы) | COUNT(заданий) за 7 дней | Ежедневно |
| **2.2.7 Индекс мастерства** | 2.2.1-2.2.6 | Weighted(интенсивность, сложность, качество) | Еженедельно |
| **2.4.2 Ступень Ученика** | 2.1.*, 2.2.*, диалоги | Алгоритм + оценка Проводника | По событию |
| **2.10.1 Профиль застревания** | 2.1.*, 2.4.2, диалоги | Классификация паттернов | По запросу |

**Принципы расчёта производных показателей:**
- Расчёт запускается **по триггеру** (новые первичные данные) или **по расписанию**
- Производные показатели **версионируются** — сохраняется история изменений
- Критические показатели (2.4.2 Ступень) требуют **подтверждения Проводником**

### 1.4. Процесс изменения данных: Use Case

```
┌────────────────────────────────────────────────────────────────────┐
│  USE CASE: Человек завершает учебную сессию                        │
└────────────────────────────────────────────────────────────────────┘

1. СОБЫТИЕ
   └─→ Человек нажимает "Завершить сессию" в LMS

2. ЗАПИСЬ ПЕРВИЧНЫХ ДАННЫХ
   └─→ LMS создаёт запись 1.3.4 (учебная сессия):
       { user_id, course_id, duration: 45min, timestamp }

3. ТРИГГЕР ПЕРЕСЧЁТА
   └─→ Централизованное хранилище получает событие
   └─→ Запускается пересчёт агрегатов для user_id

4. ОБНОВЛЕНИЕ ПРОИЗВОДНЫХ
   └─→ 2.1.1 (часы в неделю): +0.75 часа
   └─→ 2.1.2 (доля дней): пересчёт за последние 7 дней
   └─→ 2.1.6 (индекс регулярности): пересчёт если прошла неделя

5. ПРОВЕРКА ПОРОГОВ
   └─→ Если 2.1.1 достиг порога (например, 10+ часов):
       └─→ Создаётся сигнал для Проводника
       └─→ Возможен переход ступени (2.4.2)

6. УВЕДОМЛЕНИЕ ПОТРЕБИТЕЛЕЙ
   └─→ MCP-сервер получает обновлённые данные
   └─→ Проводник/Генератор могут запросить актуальное состояние
```

### 1.5. Гарантии целостности данных

| Требование | Реализация |
|------------|------------|
| **Атомарность** | Транзакции SurrealDB для связанных записей |
| **Консистентность** | Валидация по схеме модели данных |
| **Изоляция** | Версионирование производных показателей |
| **Долговечность** | Append-only log, репликация |
| **Аудит** | Полная история изменений с метаданными |

### 2.2. Алгоритм оценки ступени Ученика

**Ступени Ученика:**
| Ступень | Ключевые критерии |
|---------|-------------------|
| **Случайный** | Нет регулярного слота, хаотичное обучение |
| **Практикующий** | Появился ежедневный слот (15+ мин), пробует практики |
| **Систематический** | Слот удерживается 4+ недель, есть маршрут |
| **Дисциплинированный** | 10+ часов/неделю на протяжении 3+ месяцев |
| **Проактивный** | Понимает зачем/чему/как/когда учиться, сам инициирует |

**Процесс оценки:**

```
ВХОДНЫЕ ДАННЫЕ (из MCP-tools):
├── get_learner_core_metrics(period=4-8 недель)
│   ├── 2.1.1 Часы в неделю: 6.5
│   ├── 2.1.2 Доля дней со слотом: 0.71 (5 из 7 дней)
│   ├── 2.1.6 Индекс регулярности: 3.2 / 5.0
│   ├── 2.2.1 Интенсивность практики: 4 задания/неделю
│   ├── 2.2.7 Индекс мастерства: 2.8 / 5.0
│   └── 2.4.2 Текущая ступень: Практикующий
│
├── get_stage_history()
│   └── [Случайный → Практикующий (4 недели назад)]
│
└── get_learning_route()
    └── Текущий маршрут, статусы шагов

АЛГОРИТМ ОЦЕНКИ:
1. Проверка критериев текущей ступени
   └─→ Практикующий: слот есть, но ещё не 4 недели стабильно

2. Проверка критериев следующей ступени
   └─→ Систематический: нужно 4+ недель со слотом
   └─→ Данные показывают 3.2 недели стабильности

3. Определение профиля застревания (для Случайного)
   └─→ Не применимо (уже Практикующий)

4. Формирование вывода
   └─→ Ступень: Практикующий (подтверждено)
   └─→ До Систематического: ~1 неделя при сохранении ритма
   └─→ Риски: низкая интенсивность практики

ЗАПИСЬ РЕЗУЛЬТАТА:
└── upsert_learner_stage_evaluation(
        stage="Практикующий",
        reason="Подтверждение текущей ступени",
        metrics_snapshot={...}
    )
```

### 2.3. Формирование рекомендаций

**Типы рекомендаций по ступеням:**

| Ступень | Фокус рекомендаций | Примеры |
|---------|-------------------|---------|
| **Случайный** | Запуск минимального ритма | "Начни с 15 минут в день в одно и то же время" |
| **Практикующий** | Стабилизация слота | "Защити слот от помех, веди лог сбоев" |
| **Систематический** | Наращивание объёма | "Добавь второй слот или увеличь текущий" |
| **Дисциплинированный** | Связь с образом будущего | "Зачем тебе это? Какие системы хочешь создавать?" |
| **Проактивный** | Устойчивый стиль жизни | "Как встроить развитие в жизненный уклад?" |

**Процесс формирования рекомендации:**

```
КОНТЕКСТ:
├── Текущая ступень: Практикующий
├── Запрос человека: "Топчусь на месте"
├── Метрики: слот есть, но низкая практика
└── Узкое место: мало заданий (4/неделю при норме 7+)

СТРАТЕГИЯ ПРОВОДНИКА:
1. Валидация текущего состояния
   └─→ "Ты уже удерживаешь ежедневный слот — это прогресс"

2. Диагностика проблемы
   └─→ "Ощущение топтания может быть связано с тем, что
        слоты заняты чтением, а не практикой"

3. Конкретный шаг
   └─→ "Попробуй в каждом слоте выделять 10 минут
        на выполнение одного задания, а не только на чтение"

4. Запись в маршрут
   └─→ upsert_learning_route(
          step="Добавить практику в слот",
          metric_target="7+ заданий/неделю"
       )
```

### 3.3. Примеры персонализации по данным ЦД

**Пример 1: Случайный ученик с профилем "Хаос"**

```yaml
Входные данные ЦД:
  ступень: Случайный
  профиль_застревания: Хаос
  часы_в_неделю: 1.5
  доля_дней_со_слотом: 0.14
  интенсивность_практики: 0.5

Сгенерированное руководство:
  title: "Выход из хаоса: первые шаги к регулярности"

  раздел_1_кто_ты:
    text: "Сейчас ты в режиме случайного ученика.
           Это не плохо — это точка старта.
           Твоё обучение происходит от случая к случаю,
           без устойчивого ритма."

  раздел_2_куда:
    text: "Ближайшая цель — стать Практикующим учеником.
           Для этого нужен ежедневный слот хотя бы 15 минут."
    критерий: "7 дней подряд с учебным слотом"

  раздел_3_что:
    практики:
      - name: "Утренний минимум"
        описание: "15 минут сразу после пробуждения"

      - name: "Один абзац"
        описание: "Прочитать и выписать одну мысль"
        сложность: минимальная

  раздел_4_как:
    text: "Выбери одно время и одно место.
           Поставь напоминание.
           Не пытайся делать много — делай мало, но каждый день."
```

**Пример 2: Систематический ученик, готовящийся к переходу**

```yaml
Входные данные ЦД:
```

### 3.4. Процесс динамического обновления руководства

```
┌────────────────────────────────────────────────────────────────────┐
│  ТРИГГЕРЫ ОБНОВЛЕНИЯ ПЕРСОНАЛЬНОГО РУКОВОДСТВА                     │
└────────────────────────────────────────────────────────────────────┘

1. ИЗМЕНЕНИЕ СТУПЕНИ (2.4.2)
   └─→ Полная перегенерация руководства
   └─→ Новые цели, практики, метрики

2. ЗНАЧИМОЕ ИЗМЕНЕНИЕ МЕТРИК (2.1-2.6)
   └─→ Частичное обновление
   └─→ Корректировка целей и планов

3. ДОСТИЖЕНИЕ ЦЕЛИ МАРШРУТА
   └─→ Обновление раздела "Что делать"
   └─→ Новые практики на следующий период

4. СРЫВ / ОТКАТ
   └─→ Обновление с учётом нового состояния
   └─→ Упрощённый план восстановления

5. ПЛАНОВОЕ ОБНОВЛЕНИЕ (еженедельно)
   └─→ Синхронизация с актуальными данными ЦД
   └─→ Проверка прогресса по метрикам
```
